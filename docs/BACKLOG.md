## BL‑001 — Модуль оценки качества систем ИИ (СИИ) по ГОСТ Р 59898‑2021

**Добавлено:** 2026‑06‑30 · **Приоритет:** 🔴 высокий · **Статус:** `specced`

**Проблема.** Текущий контур оценивает качество только классических ИС по ISO 25010 /
ГОСТ Р ИСО/МЭК 25010‑2015. Для внедряемых ML/ИИ‑систем (скоринг, антифрод, OCR, чат‑боты)
нужна оценка по профильному нацстандарту **ГОСТ Р 59898‑2021**, который расширяет модель
качества ИИ‑специфичными субхарактеристиками (обучаемость, объяснимость, робастность,
контролируемость, приватность) и вводит критерий соответствия baseline ± допуски с
взвешенным интегральным показателем Q.

**Суть решения.** Второй контур оценки как **расширение** существующих сущностей
(`System.system_kind=AI`, `model_kind=GOST59898`), а не форк. Переиспользуются периоды,
значения, артефакты, экспертика, дашборды. Доработки: 4‑уровневая иерархия модели с весами,
ML‑метрики (accuracy/F1/MSE/AUC/NDCG/PSNR/SSIM), baseline+допуски, признак «СИИ».

**Объём (этапы):**
- **E1 (MVP):** seed модели 59898, признак СИИ, базовые + классификационные метрики, baseline, простой Q.
- **E2:** веса + иерархия + Q по формулам 3–8; ИИ‑субхарактеристики; регрессия/ранжирование/изображения.
- **E3:** тестовые датасеты и выбросы, паритет сред, экспертная группа + согласованность, сравнение СИИ.

**Затрагивает:** `models/system.py`, `models/metric_catalog.py`, `models/assessment.py`,
`services/calculation_engine.py` (+ новый `ai_calculation_engine.py`), новый роутер
`/api/v1/ai-assessments`, новый `constants/ai_quality_model.py` и `seed_gost59898.py`,
фронт `constants/aiQualityModel.ts` + раздел «Оценка СИИ».

**Критерии приёмки:** см. ЧАСТЬ F в [ТЗ_Модуль_Качества_СИИ_v1.md](ТЗ_Модуль_Качества_СИИ_v1.md).

**Артефакты анализа:** распарсенный официальный PDF `ГОСТ Р 59898‑2021`; сравнительная
книга моделей [ISO25010_2023_vs_GOST_25010-2015.xlsx](ISO25010_2023_vs_GOST_25010-2015.xlsx)
(вкладка «ГОСТ 59898 (ИИ)» — визуальная пара к ТЗ).

**Открытые вопросы:** источник baseline/эталонной системы (орг. процесс); роль ML‑эксперта
в [ролевой модели v12](ТЗ_Ролевая_модель_v12.md); настраиваемость номенклатуры метрик под отрасль.

## BL‑004 — Рефакторинг в доменно‑модульный монолит

**Добавлено:** 2026‑07‑03 · **Приоритет:** 🔴 высокий · **Статус:** `done` (2026‑07‑04)

**Проблема.** Backend организован по техническим слоям (`api/`, `models/`, `services/`, `core/`, `db/`);
бизнес‑домены размазаны по слоям, границ нет, всё импортирует всё. Новым модулям и интеграциям по API
(СУЗ, ТМС, ITSM, DWH) некуда лечь без размытия слоёв.

**Суть решения.** Перейти к доменно‑модульному монолиту: `modules/` (бизнес‑домены с фасадами),
`shared/` (миксины, исключения, типы, порты интеграций), `infrastructure/` (БД, Redis, celery, конфиг,
адаптеры внешних систем). Единая БД и единый `Base`. Переход — **Strangler Fig**, домены переносятся по
одному, стек рабочий, тесты зелёные на каждом шаге. Соответствие ISO/IEC 25010 (сопровождаемость) и
ISO/IEC 38500 (ИТ‑управление).

**Объём (фазы):** см. ЧАСТЬ E в [ТЗ_Модульный_Монолит_v13.md](ТЗ_Модульный_Монолит_v13.md).
Разбит на 17 подзадач (таск‑лист сессии): ТЗ → каркас → infrastructure (config/db/celery) → shared →
домены (risk→iam→systems→quality→assessment→reporting→dataio→llm) → интеграции → снятие shims.

**Прогресс (2026‑07‑04, ЗАВЕРШЕНО):** фазы 0–5 выполнены, shim‑файлы УДАЛЕНЫ (задача #18).
Все 8 доменов в `modules/`, инфраструктура в `infrastructure/` (config, database+единый Base,
redis, celery, адаптеры‑заглушки KMS/TMS/ITSM/DWH по портам `shared/ports.py`); из легаси-слоёв
остались только `api/v1/api.py` и `workers/tasks.py` (композиционные корни) и `models/audit.py`.
Регресс: **82 passed** (вкл. архитектурные тесты `tests/test_architecture.py`: легаси-пути
недоступны, фасады отдают публичный API, Dependency Rule, единый Base, celery-задачи);
холодный рестарт backend+celery зелёный; живой smoke всех доменов — 200.

**Затрагивает:** всю структуру `backend/app/**`, `alembic/env.py`, `docker-compose.yml` (путь celery),
тесты и seed‑скрипты (импорты — на этапе cleanup).

**Критерии приёмки:** ЧАСТЬ F ТЗ (регресс ≥ 31 passed на каждом шаге, `/health` ok, границы соблюдены).

## BL‑005 — Многоаспектное рассуждение LLM (Дао Тойота × ISO 25010/38500)

**Добавлено:** 2026‑07‑03 · **Приоритет:** 🟡 средний · **Статус:** `done` (2026‑07‑04)

**Проблема.** Встроенная LLM формирует заключение «в один проход» (`generate_judgment_conclusion`),
рассматривая проблему однобоко. Требуется, чтобы по карточкам мер, профсуждениям и рискам из базы
модель рассуждала **многоаспектно** и лишь затем выносила заключение руководителю.

**Суть решения.** Конвейер рассуждения на принципах Дао Тойота, переложенных на IT: Genchi Genbutsu
(идти к данным) → 5 Почему (первопричина) → Nemawashi (несколько ролевых точек зрения) → Jidoka
(встроенное качество/grounding) → Hansei (саморефлексия) → заключение. Реализация — `modules/llm/
reasoning.py` (после каркаса модулей). Манифест и обновление знаний обучения — в Obsidian (`AI/АСОК_ИС/`).

**Затрагивает:** `modules/llm/` (reasoning.py, prompts.py), `docs/LLM_TRAINING.md`, база знаний Obsidian.
**Зависит от:** BL‑004 (каркас `modules/llm`).

**Реализация (2026‑07‑04):** `modules/llm/reasoning.py` — конвейер Э0–Э7 (Генти Генбуцу → Проблема →
5 Почему → Немаваси (4 линзы: CIO/Качество/Риски/ИБ) → Дзидока → Кайдзен → Хансей → заключение ЛПР).
Дзидока-фильтры: grounding чисел, отбраковка эхо-плейсхолдеров и вырожденных повторов; каждый этап
имеет детерминированный fallback (конвейер работает и без модели). Синтез мер не запускается без
источников (карточки мер/минимизации рисков). Эндпоинт `GET /assessments/{id}/judgment-conclusion`
возвращает заключение + аудируемую трассу (`reasoning`) + `confidence`. SFT-датасет расширен полем
`reasoning` (детерминированная трасса). Покрытие: `tests/test_reasoning.py` (14 тестов).
