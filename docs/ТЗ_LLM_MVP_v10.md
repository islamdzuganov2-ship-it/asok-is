## ЧАСТЬ B. Выбор LLM для MVP (самая лёгкая и шустрая)

Критерии: on-prem (банковский контур), CPU-friendly, хороший русский, быстрый отклик, малый вес.

| Модель | Вес (q4) | Русский | Скорость CPU | Вердикт |
|--------|----------|---------|--------------|---------|
| **qwen2.5:1.5b-instruct** | ~1.0 ГБ | хороший | быстрая | **Рекомендуется (default MVP)** |
| qwen2.5:0.5b-instruct | ~0.4 ГБ | средний | очень быстрая | ультралёгкий fallback |
| llama3.2:1b-instruct | ~1.3 ГБ | средний | быстрая | альтернатива |
| gemma2:2b / phi3.5 | 1.6–2.3 ГБ | хороший | средняя | тяжелее цели MVP |

**Решение:** базовая — `qwen2.5:1.5b-instruct` (q4_K_M); ультралёгкий режим — `qwen2.5:0.5b-instruct`.
Запуск через Ollama, квантизация q4_K_M, контекст 4096, генерация ≤256 токенов.

## ЧАСТЬ C. «Обучение» LLM под требования (MVP-стратегия)

Для лёгкой модели MVP реальный fine-tune не нужен — поведение задаётся **Modelfile +
системным промптом + few-shot + параметрами**. Это и есть «обучение под требования».

### C1. Требования к LLM
1. Роль: ИТ-директор/аналитик качества банка.
2. Вход: метрики качества ИС за период (характеристика, метрика, расчётный %, уровень).
3. Выход (контракт для дашбордов): краткое резюме (2–3 предложения) + 1 рекомендация к
   действию + наиболее просевшая характеристика.
4. Стиль: строгий деловой, русский, без «воды».
5. Guardrails: **использовать только переданные цифры**, не выдумывать значения;
   при отсутствии данных — прямо сообщить об этом.
6. Детерминизм: `temperature=0.2` (управленческие выводы должны быть стабильны).

### C2. Modelfile (готов к применению — создать `backend/llm/Modelfile`)
```dockerfile
FROM qwen2.5:1.5b-instruct
PARAMETER temperature 0.2
PARAMETER top_p 0.9
PARAMETER num_ctx 4096
PARAMETER num_predict 256
PARAMETER stop "<END>"
SYSTEM """
Ты — ИТ-директор банка и аналитик качества ИС (методика МК_8.1, ISO 25010).
На вход получаешь метрики качества одной ИС за период.
Сформируй СТРОГО деловой управленческий вывод на русском языке:
1) Резюме состояния (2–3 предложения) с опорой ТОЛЬКО на переданные цифры.
2) Одну конкретную рекомендацию к действию.
3) Назови наиболее просевшую характеристику.
Запрещено выдумывать числа. Если данных нет — так и напиши.
Ответ заверши маркером <END>.
"""
```
Сборка: `ollama create asok-model -f backend/llm/Modelfile`.

### C3. Шаблон промпта (few-shot, для `generate_ai_summary_task`)
```
ИС: {system_name}. Период: {period_label}.
Метрики (характеристика | метрика | %):
{metrics_block}

Пример хорошего ответа:
«Качество ИС на низком уровне: тестируемость 25% и зрелость 32% ниже целевых.
Рекомендация: выделить ресурс на автоматизацию регрессионных тест-кейсов.
Наиболее просевшая характеристика — тестируемость. <END>»
```

### C4. Формат ответа для фронтенда
MVP: plain-text резюме в `aiInsights`. Целевой: лёгкий JSON
`{ "summary": "...", "recommendation": "...", "weakCharacteristic": "..." }`
с tolerant-парсером (если модель не вернула JSON — берём текст целиком в `summary`).

### C5. Критерии приёмки (eval)
- Ответ ≥ 50 символов, на русском, содержит рекомендацию.
- Нет чисел, которых не было во входе (проверка против списка метрик).
- Время ответа на CPU ≤ 5 c для 1.5b / ≤ 2 c для 0.5b.
- 10 эталонных кейсов проходят ручную проверку тональности/корректности.

### C6. Post-MVP (по отдельной команде)
- LoRA/QLoRA fine-tune на размеченных управленческих резюме (если потребуется стиль банка).
- RAG над методикой МК_8.1 и историей оценок.
- Кэш резюме в Redis по `(period_id, hash(metrics))`.
