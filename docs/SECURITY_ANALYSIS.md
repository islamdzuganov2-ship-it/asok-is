# АСОК ИС — анализ информационной безопасности (требования РФ)

Объект: backend (FastAPI/SQLAlchemy/Celery/JWT/LLM) + frontend (React/Redux).
Нормативная база: **152-ФЗ** (персональные данные), **ГОСТ Р 57580.1** (защита информации
финансовых организаций), **187-ФЗ** (КИИ — если ИС отнесена к значимым объектам),
требования Банка России к ИБ. ISO/IEC 25010 (характеристика «Защищённость»).

> Статусы: ✅ исправлено в этой итерации · 🟡 снижено/частично · ⏳ к доработке.

## Что исправлено в этой итерации

- **S1** — `endpoints/auth.py`: блок `DEMO_USERS` выполняется только при `settings.DEMO_MODE`.
  В проде встроенных паролей нет — аутентификация строго по БД (bcrypt, 12 раундов).
- **S2** — `config.py.security_issues()` + старт в `main.py`: при `DEMO_MODE=false` приложение
  **не стартует** с дефолтным `JWT_SECRET_KEY`/паролем БД или секретом < 32 символов.
- **S4** — `security.decode_token(token, expected_type=...)`; `/auth/refresh` принимает только
  токен с `type=refresh`, иначе 401.
- **S5** — `/excel/upload`, `/import-assessment`, `/import-workbook` требуют роль
  (TEST_ANALYST/QUALITY_MANAGER/ADMIN), `/seed-project-files` — только ADMIN.
- **S6** — проверка сигнатуры `PK\x03\x04` (OOXML/zip) дополнительно к расширению и лимиту 10 МБ.

## Положительные практики (уже были)

- Пароли — bcrypt (cost 12), хранится только хеш. ✅
- ORM-параметризация (нет конкатенации SQL) — защита от инъекций. ✅
- Загрузки: `uuid4` + `os.path.basename` — нет path traversal; лимит размера; `read_only` openpyxl. ✅
- CORS — белый список origin, не `*`. ✅
- RBAC — `require_role`, ролевые роуты на фронте. ✅

## Рекомендации к доработке (приоритет)

1. **S3/S8 (высокий):** для прод-контура отключить `DEMO_MODE`, перевести токен в `httpOnly`+`Secure`
   cookie (SameSite=Strict) вместо `localStorage`; добавить CSRF-защиту.
2. **S7:** навесить `Depends(get_current_user)` на `/reports/*` и остальные read-эндпоинты.
3. **S10:** журнал ИБ-событий (вход/выход, изменение оценок, решения по мерам, доступ к ПДн) —
   неизменяемый лог с метками времени (ГОСТ 57580, регистрация событий; основа для расследований).
4. **S11:** rate-limit и блокировка после N неудачных входов (анти-брутфорс).
5. **S9:** маскирование ПДн в логах; политика хранения и удаления (152-ФЗ ст.5, 21).
6. **ПДн (152-ФЗ):** определить состав обрабатываемых ПДн (ФИО ответственных, владельцев ИС),
   правовое основание, локализацию БД на территории РФ (ст.18), модель угроз и класс защищённости.
7. **187-ФЗ:** если ИС — значимый объект КИИ, категорирование и подключение к ГосСОПКА.
8. **LLM:** модель on-prem (данные не покидают контур) — соответствует требованиям; контролировать,
   чтобы в промпт не попадали ПДн сверх необходимого (минимизация, 152-ФЗ).
