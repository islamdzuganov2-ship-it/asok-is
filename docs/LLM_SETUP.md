# АСОК ИС — встроенная LLM (модель-агностично): установка, переключение моделей, «резервный мозг»

LLM работает **внутри процесса backend** через `llama-cpp-python` (llama.cpp). Внешних сервисов
нет. Система принимает **ЛЮБУЮ** GGUF-модель из локального каталога, **сама опрашивает её**
(архитектура, окно контекста, шаблон чата) и адаптируется. Рантайм `llama-cpp-python>=0.3`
грузит современные архитектуры (qwen2/qwen3, llama, gemma2/gemma3, phi3, mistral, stablelm и др.).

## 1. Куда класть модель
- **Каталог (хост):** `models/llm/` — сюда кладём один или несколько `*.gguf`.
- **В контейнере:** `/app/models/llm/` (монтируется read-only в `docker-compose.yml`).
- **Имя файла — любое.** Выбор файла определяет `LOCAL_LLM_MODEL_FILE`:
  - `auto` (по умолчанию) → берётся **новейший** `*.gguf` из каталога;
  - явное имя (напр. `gemma-3-12b-it-Q4_K_M.gguf`) → закрепляет конкретную модель.

## 2. Модель — без ограничений
Любая mainstream GGUF-модель грузится и опрашивается автоматически. Инструкт-модели
используют встроенный шаблон чата; для base-моделей без шаблона система откатывается к обычному
завершению текста (вывод гарантирован на любой модели). Размер — по вашему железу
(0.5B–3B комфортны на CPU; 7B–14B практичны на GPU, см. §6).

> ⚠️ Экзотический экспорт `qwen35` (Qwen3.5) может не грузиться даже в 0.3.x — это свойство
> конкретного файла, а не рантайма. Возьмите стандартный экспорт (в т.ч. обычный Qwen3).

## 3. Первый запуск (однократная пересборка под рантайм 0.3.x)
```bash
docker compose build backend celery_worker    # ставит llama-cpp-python 0.3.x
docker compose up -d
```
Проверка: `docker compose logs backend | grep LLM` → «LLM загружена: <имя> (arch=…, ctx=…)».

## 4. Переключение модели в Docker (главный сценарий)
**Пересборка образа НЕ нужна** — только положить файл и перезапустить сервисы:
1. Скопировать новый `*.gguf` в `models/llm/`.
2. Выбрать модель:
   - оставить `LOCAL_LLM_MODEL_FILE=auto` → возьмётся новейший файл; **или**
   - закрепить: `LOCAL_LLM_MODEL_FILE=<имя>.gguf` в `.env`/`docker-compose.override.yml`.
3. Применить:
   ```bash
   docker compose restart backend celery_worker
   ```
   Либо **без рестарта** (ADMIN): `POST /api/v1/reports/llm-reload`.
4. Убедиться, что переключилось:
   - `GET /api/v1/reports/llm-status`  → имя и архитектура загруженной модели;
   - `GET /api/v1/reports/llm-models`  → список всех GGUF в каталоге (какой выбран).

`docker-compose.override.yml` уже содержит пример и подхватывается автоматически.

## 5. «Резервный мозг» — обучение/настройки ВНЕ модели (переносимо между моделями)
Накопленное «обучение» не живёт внутри весов модели. Оно хранится отдельно и **переносится
при смене модели** (привязано к доменным фактам, а не к параметрам сети):
- **Каталог (хост):** `models/llm_brain/` — **writable**-том (отдельно от read-only моделей).
- **В контейнере:** `/app/models/llm_brain/`.
- **Содержимое:** `memory.jsonl` (память рассуждений → RAG-контекст следующих прогонов),
  `corpus.jsonl` (SFT-корпус для оффлайн-дообучения на любой базовой модели),
  `feedback.jsonl` (обратная связь человека: принять/поправить/отклонить),
  `profile.json` (переносимые настройки), `brain_meta.json` (какие модели пользовались мозгом).

Переключение на более умную/мощную модель → новая модель **сразу видит** всю память, корпус и
настройки. Реализация: `backend/app/modules/llm/brain.py`. Данные мозга в git не хранятся.

## 6. GPU (опционально, для крупных моделей 7B–14B)
Требуется NVIDIA GPU + драйвер + `nvidia-container-toolkit` (на Windows — через WSL2):
```bash
docker compose -f docker-compose.yml -f docker-compose.gpu.yml up -d --build
```
Оверрайд собирает образ с CUDA (`backend/dockerfile.cuda`) и включает offload всех слоёв
(`LLM_N_GPU_LAYERS=-1`). Без GPU работает стандартная CPU-сборка.

## 7. Параметры (backend/.env или config.py)
| Переменная | По умолчанию | Назначение |
|-----------|--------------|-----------|
| `LLM_ENABLED` | `true` | вкл/выкл встроенную LLM |
| `LOCAL_LLM_MODEL_DIR` | `models/llm` | каталог моделей |
| `LOCAL_LLM_MODEL_FILE` | `auto` | `auto` = автоподбор новейшего `*.gguf`; либо имя файла |
| `LLM_BRAIN_DIR` | `models/llm_brain` | каталог «резервного мозга» (writable) |
| `LLM_N_CTX` | `4096` | верхняя граница окна (ограничивается окном модели) |
| `LLM_N_GPU_LAYERS` | `0` | `0`=CPU; `-1`=offload всех слоёв на GPU |
| `LLM_CHAT_FORMAT` | `auto` | `auto`=шаблон из GGUF; иначе явный формат llama.cpp |
| `LLM_MAX_TOKENS` | `320` | длина ответа |
| `LLM_TEMPERATURE` | `0.1` | детерминизм вывода |

## 8. Честность и отсутствие галлюцинаций (сохранены)
1. **Системный промпт** — только переданные числа/факты, при отсутствии данных «данные отсутствуют».
2. **Детерминизм** — `temperature=0.1`, `top_p=0.9`.
3. **Grounding-проверка** — проценты вне входных данных → честный fallback.

## 9. Модель НЕ решает вместо системы (Rule Engine → LLM)
Вердикты считает детерминированный Python, LLM только **объясняет**:
- **Rule Engine** (`backend/app/modules/llm/gate.py`): правила `Severity>=High` (Q<41%),
  `Coverage<70%` (доля измеренных подхарактеристик), `Regression Failed` (деградация к прошлому
  периоду) — выносят вердикт и серьёзность.
- **LLM** (`reasoning.py`): получает сработавшие правила как **повод** и выдаёт
  **Объяснение · Причины · Риски · Рекомендации** строго по данным (grounding сохранён).
- Аудит: сработавшие правила видны в трассе (этапы Э0/Э4) и в ответе (`fired_rules`),
  а также в UI под заключением.

## 10. Проверка
- `GET /api/v1/reports/llm-status` → `available`, паспорт модели (`profile`), статистика мозга (`brain`).
- `GET /api/v1/reports/executive-dashboard` → `aiInsights` от LLM (при отсутствии модели — fallback).
- Тесты: `docker compose exec backend python -m pytest -q`.
