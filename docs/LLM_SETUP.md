# АСОК ИС — встроенная LLM (без Ollama): установка и запуск

LLM работает **внутри процесса backend** через `llama-cpp-python` (llama.cpp). Внешних
сервисов нет — система сама загружает GGUF-модель из локальной директории и вызывает её.

## 1. Куда положить модель
- **Путь (хост):** `models/llm/asok-model.gguf`
- **Путь в контейнере:** `/app/models/llm/asok-model.gguf` (монтируется в `docker-compose.yml`)
- **Имя файла строго:** `asok-model.gguf`

## 2. Модель
- **Рекомендуемая (MVP):** **Qwen2.5-1.5B-Instruct**, квантизация **Q4_K_M** (~1 ГБ),
  файл `models/llm/asok-model.gguf`. Стандартная архитектура GGUF `qwen2` — поддерживается
  установленным `llama-cpp-python==0.2.90`; уверенно помещается в лимит памяти контейнера (≈5.8 ГБ).
  Ультралёгкий fallback при нехватке RAM — Qwen2.5-0.5B-Instruct Q4_K_M (~0.4 ГБ).
- При наличии GPU задайте `LLM_N_GPU_LAYERS>0` (нужна сборка llama.cpp с CUDA/Metal).
- ⚠️ **Известное ограничение (DEF-01).** Файл `asok-model.gguf` версии «Qwen3.5 9B» имеет
  нестандартную архитектуру GGUF `qwen35` и **НЕ загружается** ни этой, ни любой текущей
  стабильной сборкой llama.cpp (`unknown model architecture: 'qwen35'`). В этом случае система
  деградирует на честный fallback (без падений). Для рабочего инференса замените файл на
  Qwen2.5-1.5B-Instruct Q4_K_M (см. выше). Подробности — в реестре дефектов QA от 2026-06-30.

## 3. Запуск
```bash
docker compose build backend celery_worker   # пересборка (ставит llama-cpp-python, cmake)
docker compose up -d
docker compose exec backend python -m app.scripts.seed_risk_base   # стартовая база рисков
```
Миграция БД (если используется alembic): `docker compose exec backend alembic upgrade head`
(создаст таблицу `risk_base`). В DEMO таблицы создаются автоматически (`create_all`).

## 4. Проверка
- `GET /api/v1/reports/executive-dashboard` → поле `aiInsights` теперь генерируется LLM
  (с обоснованием из базы рисков). Если файла модели нет — отдаётся корректный fallback-текст,
  приложение не падает.
- Логи: `docker compose logs backend | grep LLM` → «LLM загружена из …» либо «Файл модели LLM не найден … fallback».

## 5. Параметры (backend/.env или config.py)
| Переменная | По умолчанию | Назначение |
|-----------|--------------|-----------|
| `LLM_ENABLED` | `true` | вкл/выкл встроенную LLM |
| `LOCAL_LLM_MODEL_DIR` | `models/llm` | директория модели |
| `LOCAL_LLM_MODEL_FILE` | `asok-model.gguf` | имя файла модели |
| `LLM_N_CTX` | `4096` | размер контекста |
| `LLM_N_THREADS` | `4` | потоки CPU |
| `LLM_MAX_TOKENS` | `256` | длина ответа |
| `LLM_TEMPERATURE` | `0.2` | детерминизм вывода |

## 6. Локальный запуск без Docker (Windows)
`pip install llama-cpp-python` (нужен установленный CMake + компилятор C++; на Windows —
Visual Studio Build Tools). Положить модель в `backend/models/llm/asok-model.gguf` или задать
`LOCAL_LLM_MODEL_DIR` абсолютным путём.

## 7. Переключатель «Демо ↔ LLM» (для презентации)
- В шапке приложения — тумблер **Демо / LLM** и индикатор готовности модели (зелёный = загружена).
- **По умолчанию — «Демо» (моки):** управленческий дашборд показывает заранее подготовленные данные,
  презентация стабильна и не ждёт инференс.
- Переключение на **«LLM»**: дашборд тянет реальный `globalHealthScore` и AI-резюме от встроенной модели
  (`GET /reports/executive-dashboard`), показывает баннер «AI-резюме (встроенная LLM)». Режим запоминается
  в localStorage. Если модель не готова или backend недоступен — честный fallback/предупреждение, без падений.

## 8. Честность и отсутствие галлюцинаций
Реализовано на трёх уровнях (`backend/app/services/llm_service.py`):
1. **Системный промпт** — строгие правила: использовать только переданные числа/факты, не домысливать,
   при отсутствии данных писать «данные отсутствуют».
2. **Детерминизм** — `temperature=0.1`, `top_p=0.9`.
3. **Grounding-проверка пост-фактум** — если в ответе появляются проценты, которых НЕ было во входных
   данных, ответ признаётся недостоверным и заменяется честным резюме строго по расчётным метрикам.

## 9. «Обучение» под требования
Поведение задаётся системным промптом в `backend/app/services/llm_service.py` (`SYSTEM_PROMPT`):
роль ИТ-директора/аналитика качества, формат «резюме + рекомендация + просевшая характеристика»,
запрет на выдуманные числа. Для кастомной модели можно собрать свою через Ollama Modelfile и
сконвертировать в GGUF, либо дообучить (LoRA) и квантовать — детали в `docs/ТЗ_LLM_MVP_v10.md`.
