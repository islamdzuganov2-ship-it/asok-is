# АСОК ИС — ТЗ v9: Управленческий дашборд (C-Level) и дашборд Менеджера по качеству

> Документ обогащает методику МК_8.1 (ISO 25010, 28 метрик, 9 характеристик) требованиями
> к двум ролевым интерфейсам и фиксирует решения по оптимизации кода из ревью.

---

## ШАГ 1. Системная аналитика (as-is)

**Стек:** FastAPI + SQLAlchemy(async) + Alembic / React 18 + AntD 5 + ECharts 5 + RTK Query / Docker Compose.

**Роли (RBAC):** `TEST_ANALYST` (ввод метрик), `QUALITY_MANAGER` (суждения, утверждение), `ADMIN`
(справочники), `EXECUTIVE/CTO` (дашборд, AI-резюме).

**Data-flow:** Метрика (A,B) → `calculation_engine` (Sₓ = 1 − A/B или A/B для INVERSE) → уровень
качества (6 градаций) → экспертное суждение (append-only) → агрегация в дашборд.

**As-is фронтенд:** единый `DashboardPage.tsx` для всех ролей; `ExpertReviewPage` + `ExpertJudgmentModal`
для суждений. Три ролевых дашборда из плана `АСОК_ИС_три_дашборда_рефакторинг` **не реализованы**.

## ШАГ 2. Проблемные зоны исходного ТЗ

| # | Проблема | Влияние |
|---|----------|---------|
| P1 | Один дашборд на все роли — топ-менеджмент видит «сырые» метрики аналитика | Перегруз, не читается C-Level |
| P2 | Нет петли «суждение менеджера → одобрение топ-менеджментом» | Меры не доходят до ЛПР |
| P3 | Суждение не привязано к под-характеристике и не порождает задачу/риск | Нет трассируемости |
| P4 | Нет требования к «нативной» подаче проблемы + рекомендации к действию | Дашборд = отчёт, а не инструмент решений |
| P5 | Кричащая палитра (#f5222d/#52c41a) против требования «спокойные тона» | Не usability для C-Level |
| P6 | Рассинхрон ключа токена (`token` vs `asok_access_token`) | Дашборд не грузится |
| P7 | Бэкенд-эндпоинты `/dashboard/{role}`, `/metrics/{id}/judgment` отсутствуют | Фронт нечем питать |

## ШАГ 3. Требования к ролевым дашбордам (to-be)

### 3.1 Управленческий дашборд (CIO/CEO/CTO)
- **R1.1** Общий индекс качества ИТ-ландшафта — Gauge со спокойной шкалой (приглушённые
  зелёный/янтарный/терракот), подпись уровня словом.
- **R1.2** «ТОП-3 проблемных ИС» — карточки с AI-резюме, RAG-процентом и **одной рекомендацией к действию**.
- **R1.3** Тепловая карта ИС × характеристики — RAG-маркеры, спокойные тона.
- **R1.4** Статус технического долга (Risks Burndown) — прогресс устранения.
- **R1.5** **По клику** на проблемную ИС/ячейку — компактное модальное окно:
  «Кто виноват» (владелец/ответственный) · «С кого спрашивать» (эскалация) ·
  «Рекомендуемые действия» · связанные меры менеджера, ожидающие одобрения.
- **R1.6** Минимум текста и цвета; интерактив вместо простыни данных.

### 3.2 Дашборд Менеджера по качеству
- **R2.1** Выбор ИС и характеристики/под-характеристики, Gauge характеристики.
- **R2.2** Таблица метрик: Метрика · Расчётный % · Уровень (RAG).
- **R2.3** Модал **«Профессиональное суждение»**: метрика · обоснование (текст) ·
  чекбокс «Создать карточку риска» → Риск · Ответственный · Срок · [Применить корректировку].
- **R2.4** Постановка задачи исходя из под-характеристики и требований.
- **R2.5** Все задачи/суждения уходят топ-менеджменту как **меры/намерения со статусом
  «ожидает одобрения»**; ЛПР видит, **что от него ждут и почему** (контекст: метрика, уровень, риск, обоснование).

### 3.3 Governance-петля (связывает обе роли)
`Менеджер создаёт меру (PENDING_APPROVAL)` → `Executive видит в «Ожидают одобрения»` →
`Одобрить/Отклонить` → статус обновляется у обеих ролей.
Реализуется слоем `governanceSlice` (Redux + localStorage), интерфейс совместим с будущим
`POST /api/v1/metrics/{id}/judgment` и `PATCH /governance/proposals/{id}`.

## ШАГ 4. Код-ревью (ключевые находки)

| Severity | Файл | Находка | Решение |
|----------|------|---------|---------|
| HIGH | `DashboardPage.tsx:52` | токен `asok_access_token` ≠ `authSlice` `token` | Единый ключ `token` |
| HIGH | `apiSlice.ts:120` | fallback `localStorage('token')` дублирует логику | Централизовать в `prepareHeaders` |
| MED | `src/app/App.tsx` | «мусорный» файл-фрагмент, не используется (`main.tsx`→`./App`) | Удалить из сборки |
| MED | `DashboardPage.tsx` | ручной `echarts.init` + resize-листенеры | Перейти на `echarts-for-react` (уже в deps) |
| MED | палитра | насыщенные цвета | Токены спокойной RAG-палитры |
| LOW | роли | строковые литералы ролей размазаны | Константы `ROLES` |

## ШАГ 5. Решения по оптимизации (внесены в ТЗ)

- **O1** Ленивая загрузка ролевых дашбордов (`React.lazy` + `Suspense`).
- **O2** `echarts-for-react` вместо ручного init — меньше кода, авто-resize.
- **O3** Единая спокойная RAG-палитра в `theme/ragPalette.ts` (переиспользование).
- **O4** Governance-слой с persist в localStorage — работает без бэкенда, интерфейс под будущий API.
- **O5** RTK cache-invalidation: одобрение меры инвалидирует `executive` + `manager`.
- **O6** Селекторы дашборда по роли в `AppLayout` (один источник навигации).
