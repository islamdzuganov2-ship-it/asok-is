# АСОК ИС — ТЗ v13: Доменно-модульный монолит (Modular Monolith)

> Документ описывает **целевую архитектуру** backend АСОК ИС и **инкрементальный план перехода**
> к ней из текущего слоевого монолита. Переход выполняется по паттерну **Strangler Fig**:
> новая структура вводится рядом со старой, домены переносятся по одному, Docker-стек остаётся
> рабочим, а автотесты — зелёными на каждом шаге. Связано: [BACKLOG](BACKLOG.md) (BL‑004),
> [Ролевая модель v12](ТЗ_Ролевая_модель_v12.md), [Модуль СИИ v1](ТЗ_Модуль_Качества_СИИ_v1.md).

---

## 0. Цель и драйверы

**Проблема.** Backend организован по **техническим слоям** (`api/`, `models/`, `schemas/`,
`services/`, `core/`, `db/`). Бизнес-домены (реестр ИС, оценка качества, база рисков, меры/
governance, LLM, отчётность, импорт данных) «размазаны» по этим слоям. Следствия:

- нет границ между доменами — любой файл импортирует любой (высокая связность, «большой ком грязи»);
- изменение одного домена задевает файлы во всех слоях (низкая **модифицируемость**, ISO 25010);
- новым модулям (СИИ по ГОСТ Р 59898, будущие домены) и **интеграциям по API** (СУЗ, ТМС, ITSM,
  хранилища данных) **некуда лечь** без размытия слоёв ещё сильнее;
- тестируемость домена в изоляции затруднена (нет чёткого фасада/контракта).

**Драйверы решения (что закладываем на будущее):**
1. Подключение новых бизнес-модулей без переписывания существующих (напр., СИИ — [BL‑001](BACKLOG.md)).
2. Интеграция по API с внешними системами: **СУЗ** (система управления знаниями/KMS),
   **ТМС** (система управления тестированием, напр. TestOps/qTest), **ITSM** (Jira Service
   Management / ServiceNow), **хранилища данных (DWH)** — как приём данных, так и **выгрузка
   собственного анализа** в DWH.
3. Соответствие **ISO/IEC 25010** (качество ИС, подхарактеристики сопровождаемости) и
   **ISO/IEC 38500** (корпоративное управление ИТ).

**Решение.** Перейти к **доменно-модульному монолиту**: единое приложение и единая БД, но код
разбит на автономные бизнес-**модули** с явными границами и контрактами, поверх тонкого слоя
**infrastructure** и общего **shared**-ядра. Микросервисы избыточны на текущем масштабе; модульный
монолит даёт изоляцию доменов при простоте развёртывания и возможность «выломать» модуль в сервис
позже, если понадобится.

## ЧАСТЬ A. Анализ текущей архитектуры (as-is)

### A1. Структура каталогов
```
backend/app/
  api/
    deps.py                 # get_db (дубль), get_current_user, require_role
    v1/
      api.py                # сборка роутеров
      reports.py            # дашборды + measures-analytics (домен reporting)
      risk_base.py          # CRUD базы рисков (домен risk)
      excel_upload.py       # загрузка Excel (домен dataio)
      endpoints/{auth,systems,assessments,metrics}.py
  core/       config.py, database.py (engine/session/get_db дубль), security.py
  db/         base.py (единый Base + регистрация всех моделей), session.py
  models/     user, system, metric_catalog, assessment, matrices, risk_base, audit, base_mixin
  schemas/    auth, assessment, metric, risk_base, common
  services/   llm_service, calculation_engine, excel_importer
  constants/  quality_model.py (модель 25010)
  scripts/    seed_* (демо-данные, каталог метрик)
  workers/    tasks.py (celery: parse_excel, generate_ai_summary, cache_invalidate)
  main.py     # точка сборки FastAPI
```

### A2. Домены, «размазанные» по слоям
| Бизнес-домен | Где сейчас лежит (файлы в разных слоях) |
|---|---|
| **IAM** (польз./аутентификация) | `models/user`, `core/security`, `schemas/auth`, `api/deps`, `api/v1/endpoints/auth` |
| **Systems** (реестр ИС) | `models/system`, `api/v1/endpoints/systems` |
| **Quality** (модель качества) | `models/metric_catalog`, `constants/quality_model`, `services/calculation_engine`, `api/v1/endpoints/metrics` |
| **Assessment** (оценки, профсуждения) | `models/assessment`, `schemas/assessment`, `api/v1/endpoints/assessments` |
| **Risk** (база рисков) | `models/risk_base`, `schemas/risk_base`, `api/v1/risk_base` |
| **Reporting** (дашборды, матрицы) | `models/matrices`, `api/v1/reports`, часть `schemas/assessment` |
| **DataIO** (импорт Excel) | `services/excel_importer`, `api/v1/excel_upload`, `workers/tasks:parse_excel` |
| **LLM** (встроенная модель) | `services/llm_service`, `scripts/export_llm_dataset`, `llm/train_lora`, `workers/tasks:generate_ai_summary` |

### A3. Граф зависимостей — болевые точки
- **Единый declarative `Base`** в `app/db/base.py` импортируется всеми моделями и регистрирует их
  для Alembic/`create_all`. Это **линчпин** — трогать осторожно (нельзя расколоть `metadata` на два `Base`).
- **`app.core.config.settings`** и **`app.core.database`** (engine, `AsyncSessionLocal`, `get_db`) —
  вторая и третья опоры, импортируются повсюду.
- **Дублирование**: `get_db` определён и в `core/database.py`, и в `api/deps.py`; ещё `get_session`
  в `db/session.py`. Три разных места одной сессии.
- **Кросс-доменные связи ORM** идут по строкам: `relationship("System")`, `ForeignKey("systems.id")` —
  это хорошо: жёстких import-циклов между моделями нет, связи резолвятся по реестру классов.
- **Некуда класть интеграции**: у внешних коннекторов (СУЗ/ТМС/ITSM/DWH) нет своего слоя.

### A4. Что уже хорошо (сохраняем)
- Полностью async (SQLAlchemy async + FastAPI), пул соединений настроен.
- Конфиг только из env (`pydantic-settings`), есть контроль небезопасной конфигурации на старте
  (ГОСТ Р 57580, 152‑ФЗ) — оставляем в `infrastructure/config`.
- `constants/quality_model` намеренно не импортирует ORM (нет циклов) — образец для `shared`.
- Grounding-контроль LLM (детерминизм + пост-проверка чисел) — переносим как есть.
- Код примонтирован томом (`./backend:/app`, `--reload`) → перенос файлов виден вживую без пересборки.

## ЧАСТЬ B. Целевая архитектура (Modular Monolith)

### B1. Принципы
1. **Доменная нарезка вместо слоевой.** Верхний уровень — бизнес-домены (`modules/*`), а не техслои.
2. **Слоистость внутри модуля.** Каждый модуль самодостаточен: `router → service → repository/models`,
   `schemas` (контракт API), `domain` (правила). Внутри — зависимость только «вниз».
3. **Зависимость внутрь (Dependency Rule).** `modules` → `shared`/`infrastructure`; `shared` и
   `infrastructure` **не** знают про `modules`. Домены **не** импортируют внутренности друг друга —
   только публичный фасад (`modules/<x>/__init__.py`) или общий контракт в `shared`.
4. **Фасад модуля.** Каждый модуль экспортирует наружу узкий публичный API (сервисные функции,
   Out-схемы, роутер). Всё остальное — приватная реализация.
5. **Инверсия зависимостей для внешнего мира.** Внешние системы (СУЗ/ТМС/ITSM/DWH) описаны
   **портами** (Protocol/ABC) в `shared`/модуле, реализованы **адаптерами** в `infrastructure/integrations`.
   Домен зависит от порта, не от конкретного клиента.

### B2. Целевая структура каталогов
```
backend/app/
  main.py                         # composition root: сборка app, монтаж роутеров модулей
  bootstrap.py                    # регистрация всех модулей (роутеры, модели, стартовые хуки)

  modules/                        # ← БИЗНЕС-ДОМЕНЫ
    iam/            router.py service.py models.py schemas.py security.py deps.py
    systems/        router.py service.py models.py schemas.py
    quality/        router.py service.py models.py schemas.py quality_model.py calculation.py
    assessment/     router.py service.py models.py schemas.py
    risk/           router.py service.py models.py schemas.py
    reporting/      router.py service.py models.py schemas.py
    dataio/         router.py service.py importer.py tasks.py
    llm/            service.py prompts.py dataset.py reasoning.py   # reasoning.py — задача 1 (Дао‑Тойота)
    ai_quality/     (задел под СИИ, BL‑001)

  shared/                         # ← ОБЩЕЕ ЯДРО (без бизнес-логики домена)
    db.py            # TimestampMixin, SoftDeleteMixin (общие миксины)
    exceptions.py    # базовые доменные исключения (NotFound, Conflict, DomainError)
    types.py         # общие типы/алиасы, перечисления кросс-домена
    ports.py         # протоколы внешних интеграций (KMS/TMS/ITSM/DWH) — контракты

  infrastructure/                 # ← ТЕХНИЧЕСКИЙ ФУНДАМЕНТ
    config.py        # Settings (env), security_issues()
    database.py      # engine, AsyncSessionLocal, get_db, Base (ЕДИНЫЙ), model registry
    redis.py         # redis-клиент/фабрика
    workers.py       # celery_app
    integrations/    # адаптеры внешних систем (реализации портов из shared/ports)
      kms/   (СУЗ)   tms/   (ТМС)   itsm/  (ITSM)   dwh/   (хранилище: приём + выгрузка анализа)
```

### B3. Карта модулей (ответственность и фасад)
| Модуль | Ответственность | Публичный фасад (наружу) |
|---|---|---|
| `iam` | Пользователи, аутентификация (JWT), роли, SoD | `get_current_user`, `require_role`, `router` |
| `systems` | Реестр ИС, критичность, ЖЦ | `router`, `SystemOut`, `get_system(...)` |
| `quality` | Модель 25010, каталог метрик, расчёт X/уровня | `router`, `calculate_metric`, `map_to_level`, `QUALITY_MODEL` |
| `assessment` | Периоды, значения, профсуждения, экспертика | `router`, `AssessmentOut`, `list_judgments(...)` |
| `risk` | Сквозная база рисков (grounding для LLM) | `router`, `search_risks(...)`, `RiskBaseOut` |
| `reporting` | Дашборды (exec/manager), матрицы Excel, аналитика мер | `router`, `build_executive_dashboard(...)` |
| `dataio` | Импорт Excel, будущий приём из внешних источников | `router`, `import_workbook(...)`, celery-задачи |
| `llm` | Инференс, grounding, промпты, датасет, **многоаспектное рассуждение** | `generate_judgment_conclusion(...)`, `model_info()` |
| `ai_quality` | Оценка СИИ по ГОСТ Р 59898 (задел) | — |

### B4. Правила зависимостей
```
          ┌─────────────────────────────────────────────┐
          │                   modules/                   │
          │  iam  systems  quality  assessment  risk     │
          │  reporting  dataio  llm  ai_quality          │
          └───────┬───────────────────────┬─────────────┘
                  │ (только через фасады)  │ (порты из shared)
                  ▼                        ▼
          ┌───────────────┐        ┌────────────────────────┐
          │    shared/    │        │   infrastructure/      │
          │ db exceptions │◄───────│ config db redis        │
          │ types ports   │        │ workers integrations/  │
          └───────────────┘        └────────────────────────┘
```
- Разрешено: `modules/*` → `shared`, `infrastructure`; `infrastructure` → `shared`.
- Запрещено: `shared`/`infrastructure` → `modules`; прямой импорт приватных модулей чужого домена.
- Кросс-доменные ORM-связи — по строковым ссылкам (`ForeignKey("systems.id")`,
  `relationship("System")`), как сейчас; классы регистрируются в едином `Base` через реестр моделей.
- Кросс-доменные бизнес-вызовы — только через публичный фасад соседнего модуля (напр. `reporting`
  вызывает `risk.search_risks(...)` и `llm.generate_...`, а не лезет в их модели напрямую — на этапе
  cleanup).

### B5. Слой интеграций (СУЗ / ТМС / ITSM / DWH)
**Порты** (контракты) — в `shared/ports.py`, **адаптеры** — в `infrastructure/integrations/*`.
Домен объявляет потребность через порт; конкретный клиент внедряется зависимостью.

| Внешняя система | Порт (протокол) | Направление | Назначение |
|---|---|---|---|
| **СУЗ (KMS)** | `KnowledgeSource` | ⇄ | Приём статей/регламентов как grounding-контекста для LLM; публикация заключений/мер обратно в базу знаний |
| **ТМС (TMS)** | `TestManagementSource` | → | Приём результатов тестов/покрытия для метрик надёжности/тестируемости (25010) |
| **ITSM** | `IncidentSource` | → | Приём инцидентов/проблем/изменений для метрик доступности, MTTR, дефектности |
| **DWH (хранилище)** | `DataWarehouseSink` / `DataWarehouseSource` | ⇄ | **Приём** сырых данных для оценок; **выгрузка** рассчитанного анализа/дашбордов/заключений в витрины |

Принципы слоя: идемпотентный приём (upsert по внешнему ключу), устойчивость к недоступности
(таймауты + деградация, как у LLM), маппинг «внешняя сущность → домен» в адаптере, а не в домене;
секреты/эндпоинты — через `infrastructure/config`.

### B6. Данные и миграции
- **Единый `Base`/`metadata`** остаётся один (в `infrastructure/database.py`). Модели модулей
  импортируют его оттуда. Центральный **реестр моделей** (`infrastructure/database.py` или
  `bootstrap.py`) импортирует `models.py` каждого модуля, чтобы Alembic видел все таблицы.
- `alembic/env.py` переключается на `infrastructure.database:Base` (после переноса; до этого работает
  через shim). Существующие миграции (`001…005`) не переписываются.
- Одна БД, транзакции в пределах запроса; кросс-модульные инварианты — на уровне сервиса-оркестратора.

## ЧАСТЬ C. Соответствие стандартам

### C1. ISO/IEC 25010 — подхарактеристики сопровождаемости (прямой эффект рефакторинга)
| Подхарактеристика | Как обеспечивается целевой архитектурой |
|---|---|
| **Модульность** | Домены — автономные модули с границами и фасадами |
| **Повторное использование** | `shared`-ядро и порты интеграций переиспользуются модулями |
| **Анализируемость** | Локализация изменения в одном модуле; явный граф зависимостей |
| **Модифицируемость** | Правка домена не задевает соседние (Dependency Rule) |
| **Тестируемость** | Модуль тестируется через фасад с подменой портов (моки СУЗ/ТМС/ITSM/DWH) |

### C2. ISO/IEC 38500 — 6 принципов корпоративного управления ИТ
| Принцип | Отражение в архитектуре |
|---|---|
| **Ответственность** | Владелец на модуль; SoD в `iam` (ролевая модель v12) |
| **Стратегия** | Задел под модули/интеграции без переписывания (соответствие драйверам §0) |
| **Приобретение** | Порты/адаптеры → внешние системы подключаемы по контракту, без vendor lock-in |
| **Производительность** | Единый стек/БД, кэш Redis, деградация внешних вызовов |
| **Соответствие** | Слой config с проверкой секретов (ГОСТ Р 57580, 152‑ФЗ); аудит в `iam`/`shared` |
| **Человеческий фактор** | Границы модулей = граница зон ответственности команд |

### C3. Безопасность (сохраняется)
Контроль небезопасной конфигурации на старте (JWT-секрет, пароль БД) переезжает в
`infrastructure/config`; поведение не меняется.

## ЧАСТЬ D. Сравнение as-is ↔ target
| Критерий | As-is (слоевой) | Target (модульный) |
|---|---|---|
| Верхний уровень | Техслои | Бизнес-домены |
| Границы домена | Нет | Явные (фасад + правила зависимостей) |
| Радиус изменения | Через все слои | В пределах модуля |
| Новый модуль | Правки во всех слоях | Новая папка `modules/<x>` + монтаж |
| Внешние интеграции | Некуда класть | `infrastructure/integrations` по портам |
| Тест домена в изоляции | Трудно | Через фасад + моки портов |
| Развёртывание | Один процесс | Один процесс (без изменений) |
| Готовность к «выломать в сервис» | Низкая | Высокая (границы уже есть) |

## ЧАСТЬ E. План миграции (Strangler Fig, инкрементально)

> Правило каждого шага: стек остаётся рабочим, автотесты зелёные (**база: 31 passed**),
> старые пути импорта продолжают работать через **re-export shim**, пока ссылки не переведены.

**Фаза 0 — Документы и каркас** *(эта сессия)*
- E0.1 ТЗ (этот документ) + BL‑004 в [BACKLOG](BACKLOG.md).
- E0.2 Каркас `modules/`, `shared/`, `infrastructure/` + `ARCHITECTURE.md` (конвенции).

**Фаза 1 — Infrastructure** *(эта сессия — начало)*
- E1.1 `config` → `infrastructure/config.py`; `app/core/config.py` — shim.
- E1.2 `database`+`Base` → `infrastructure/database.py`; `app/core/database.py`, `app/db/base.py` — shims.
- E1.3 `celery`/`redis` → `infrastructure/workers.py`, `infrastructure/redis.py`; `app/workers/tasks.py` — shim.

**Фаза 2 — Shared** — миксины, исключения, типы, порты интеграций (`shared/ports.py`).

**Фаза 3 — Домены по одному** (в порядке связности, снизу вверх):
`risk` (пилот) → `iam` → `systems` → `quality` → `assessment` → `reporting` → `dataio` → `llm`.
Каждый: перенос `models/schemas/router/service` в `modules/<x>/`, shims на старых путях, монтаж роутера через модуль, прогон тестов.

**Фаза 4 — Интеграции** — `infrastructure/integrations/{kms,tms,itsm,dwh}` адаптеры-заглушки по портам.

**Фаза 5 — Cleanup** — перевод `alembic/env.py`, `bootstrap`, тестов и скриптов на новые пути;
удаление shims; финальный регресс + smoke API.

Соответствие таск-листу: E0.1→#1, E0.2→#3/BL‑004→#2, E1.x→#4–#6, Фаза2→#7, Фаза3→#8–#15,
Фаза4→#16, Фаза5→#17.

## ЧАСТЬ F. Критерии приёмки
1. **Регресс не падает:** на каждом шаге `docker compose exec backend pytest -q` = ≥ 31 passed.
2. **Стек жив:** `GET /health` = ok; `GET /api/v1/reports/llm-status` отвечает; дашборды грузятся.
3. **Обратная совместимость:** старые импорты работают до фазы cleanup (shims), Alembic видит все модели.
4. **Границы соблюдены:** `shared`/`infrastructure` не импортируют `modules`; домены общаются через фасады
   (проверяется грепом/линтом на этапе cleanup).
5. **Интеграции по контракту:** домены зависят от портов из `shared/ports`, не от клиентов из `infrastructure`.
6. **Документация синхронна:** `ARCHITECTURE.md` и этот ТЗ отражают фактическую структуру.

## ЧАСТЬ G. Риски и откат
| Риск | Митигация |
|---|---|
| Раскол `Base`/`metadata` на два (модели «пропадают» для Alembic) | `Base` — ровно один, в `infrastructure/database`; shims только **ре-экспортируют** его |
| Слом пути celery (`-A app.workers.tasks.celery_app`) | Shim сохраняет символ `celery_app` по старому пути до правки compose |
| Циклические импорты при переносе | Кросс-модель — строковые ссылки; фасады тонкие; порты в `shared` |
| Незамеченная регрессия | Тесты после КАЖДОГО шага; шаги маленькие и обратимые (git) |
| Долгий незавершённый рефакторинг | Strangler: система рабочая на любом промежуточном шаге |

## ADR‑001. Modular Monolith vs альтернативы
**Контекст.** Растущий контур (новые модули + интеграции СУЗ/ТМС/ITSM/DWH), одна команда, один стек.
**Рассмотрено:** (а) оставить слоевой монолит — отклонено (не масштабируется по доменам);
(б) микросервисы — отклонено (оверинжиниринг: распределённые транзакции, ops-нагрузка на текущем масштабе);
(в) **доменно-модульный монолит** — **принято**.
**Решение.** Модульный монолит: изоляция доменов + простота одного развёртывания; границы позволяют
позже «выломать» модуль в сервис. **Следствия:** вводим `modules/shared/infrastructure`, порты для
внешних систем, strangler-миграцию. **Статус:** accepted (2026‑07‑03).
