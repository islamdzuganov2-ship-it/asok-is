# Полная пошаговая инструкция — прямой хостинг asokis.ai с вашего ПК (без туннеля)

Результат: `https://asokis.ai` отдаётся **напрямую вашим ПК**. Cloudflare работает только как DNS
(серое облако), трафик через него НЕ идёт — «без прокладок». TLS-сертификат ваш ПК получает сам
(Let's Encrypt через Caddy).

Ваши данные (подставлены заранее):
- Домен: `asokis.ai`
- Публичный IP (как видит интернет): `149.88.28.129`
- LAN-адрес ПК: `192.168.1.97`
- Роутер (шлюз): `192.168.1.1`

> Прямой хостинг сложнее и рискованнее туннеля. Он сработает ТОЛЬКО если ваш провайдер даёт
> «белый» (публичный) IP и не блокирует входящие 80/443. Сначала рекомендуется пройти раздел
> **«Проверка достижимости»** (шаг 7) — если порт снаружи закрыт, прямой путь невозможен,
> используйте `docs/SETUP_CLOUDFLARE_TUNNEL.md`.

## 0. Предусловия
1. Docker Desktop запущен. Проект в `C:\Users\adiga\projects\asok-is`.
2. Базовый стенд работает локально: `docker compose up -d` → `http://localhost:3000`.
3. Домен `asokis.ai` должен стать **Active** в Cloudflare (смена NS у регистратора на
   `desi.ns.cloudflare.com` / `patrick.ns.cloudflare.com`, 1–24 ч). См. шаг 1 гайда по туннелю.

## 1. Зафиксировать локальный IP ПК (DHCP-резервация)
Чтобы `192.168.1.97` не «уехал» после перезагрузки:
1. Войдите в роутер: браузер → `http://192.168.1.1` (логин/пароль на наклейке роутера).
2. Раздел **DHCP → Address Reservation / Static Leases** (название зависит от модели).
3. Привяжите MAC-адрес ПК к IP `192.168.1.97`.
   - MAC узнать в PowerShell: `Get-NetAdapter | Where-Object Status -eq 'Up' | Select Name,MacAddress`
4. Сохраните.

---

## 2. Открыть порты во встроенном брандмауэре Windows
Запустите **PowerShell от имени администратора** (правый клик → «Запуск от имени администратора»):
```powershell
New-NetFirewallRule -DisplayName "ASOK HTTP"  -Direction Inbound -Protocol TCP -LocalPort 80  -Action Allow
New-NetFirewallRule -DisplayName "ASOK HTTPS" -Direction Inbound -Protocol TCP -LocalPort 443 -Action Allow
```
Проверка, что правила создались:
```powershell
Get-NetFirewallRule -DisplayName "ASOK*" | Select DisplayName,Enabled,Direction,Action
```
Удалить позже (откат): `Get-NetFirewallRule -DisplayName "ASOK*" | Remove-NetFirewallRule`

> Убедитесь, что порт 80 не занят (IIS/«Службой веб-публикации»). Проверка:
> `Get-NetTCPConnection -State Listen | ? LocalPort -in 80,443`. Если занят IIS —
> остановите его: `Stop-Service W3SVC` (от админа).

---

## 3. Проброс портов на роутере (Port Forwarding)
1. Роутер `http://192.168.1.1` → раздел **Port Forwarding / Virtual Server / NAT**.
2. Создайте два правила (протокол TCP):

   | Внешний порт | Внутренний IP   | Внутренний порт |
   |--------------|-----------------|-----------------|
   | 80           | 192.168.1.97    | 80              |
   | 443          | 192.168.1.97    | 443             |

3. Сохраните и при необходимости перезагрузите роутер.

> Если в роутере есть «DMZ host» — НЕ используйте его (открывает все порты). Только проброс 80/443.

---

## 4. DNS-записи в Cloudflare (режим DNS-only — серое облако)
> Делать после того, как `asokis.ai` стал **Active**.
1. Панель Cloudflare → домен `asokis.ai` → **DNS → Records → Add record**.
2. Запись 1:
   - **Type**: `A`
   - **Name**: `@`  (это корень `asokis.ai`)
   - **IPv4 address**: `149.88.28.129`
   - **Proxy status**: кликните по облаку, чтобы оно стало **серым (DNS only)** — это критично:
     оранжевое = трафик через Cloudflare (прокладка), серое = напрямую к вам.
   - **Save**
3. Запись 2 (для `www`):
   - **Type**: `A` · **Name**: `www` · **IPv4**: `149.88.28.129` · **Proxy: DNS only (серое)** · Save

Проверка резолва (через пару минут), PowerShell:
```powershell
Resolve-DnsName asokis.ai -Server 1.1.1.1
```
Должен вернуть `149.88.28.129`.

## 5. (Важно при домашнем интернете) Авто-обновление IP — DDNS
Домашний публичный IP обычно **динамический** и со временем меняется — тогда A-запись «протухнет».
Решение: контейнер DDNS, который сам обновляет A-запись в Cloudflare.

1. Создайте **API-токен** Cloudflare: панель → значок профиля → **My Profile → API Tokens →
   Create Token → шаблон «Edit zone DNS»**. Zone Resources: Include → Specific zone → `asokis.ai`.
   Создать → скопировать токен.
2. В `.env` проекта:
   ```
   CLOUDFLARE_API_TOKEN=<скопированный токен>
   DDNS_DOMAINS=asokis.ai,www.asokis.ai
   ```
3. Запустить:
   ```powershell
   docker compose --profile ddns up -d
   docker compose logs -f ddns
   ```
   Контейнер будет периодически выставлять текущий IP в A-записи (PROXIED=false — остаётся серое облако).

> Альтернатива: встроенный DDNS многих роутеров (раздел Dynamic DNS) с провайдером Cloudflare.

## 6. Запустить прямой веб-сервер (Caddy, авто-TLS)
```powershell
docker compose --profile direct up -d
docker compose logs -f webserver
```
Caddy запросит у Let's Encrypt сертификат для `asokis.ai` (нужен доступ снаружи к порту 80 —
проверка ACME HTTP-01). В логах ожидайте строки про `certificate obtained` / `serving`.

> Сертификат выдастся ТОЛЬКО когда выполнены: домен Active + DNS A-запись на ваш IP +
> проброс 80 + firewall. Пока чего-то нет — Caddy будет повторять попытки.

---

## 7. Проверка достижимости снаружи (сделайте это пораньше!)
Самый частый стоп-фактор — провайдер не пускает входящие (CGNAT или блок 80/443).
1. С телефона по **мобильному интернету** (не по домашнему Wi-Fi!) откройте:
   `http://149.88.28.129` — если что-то ответило (хотя бы редирект/ошибка Caddy), порт открыт.
2. Или онлайн-чекер портов: проверьте `149.88.28.129` порт `443` и `80`
   (например, ifconfig.co/port или canyouseeme.org).
3. Результат:
   - **порт открыт** → продолжайте, всё заработает после получения сертификата;
   - **порт закрыт** → провайдер блокирует входящие / CGNAT. Прямой хостинг невозможен —
     переходите на `docs/SETUP_CLOUDFLARE_TUNNEL.md` (туннель работает без белого IP).

Финальная проверка: откройте `https://asokis.ai` в браузере — приложение должно открыться
с валидным замком (TLS).

---

## 8. Управление
- Остановить публичный веб-сервер (локальный `:3000` останется):
  ```powershell
  docker compose stop webserver
  ```
- Остановить DDNS: `docker compose stop ddns`
- Полностью убрать: `docker compose rm -sf webserver ddns`
- Откатить firewall: `Get-NetFirewallRule -DisplayName "ASOK*" | Remove-NetFirewallRule` (от админа)
- Убрать проброс портов — в панели роутера.

## 9. Диагностика проблем
| Симптом | Причина / решение |
|--------|-------------------|
| `https://asokis.ai` не открывается, порт снаружи закрыт | CGNAT/блок провайдера. Прямой путь невозможен → туннель. |
| Caddy не выдаёт сертификат | Нет доступа к порту 80 снаружи, или DNS ещё не указывает на ваш IP, или зона не Active. Проверьте шаги 3–4 и резолв `asokis.ai`. |
| Открылась чужая страница / роутер | Неверный проброс или оранжевое облако в DNS. Включите серое (DNS only). |
| Сайт был доступен, потом пропал | Сменился домашний IP. Включите DDNS (шаг 5) или обновите A-запись вручную. |
| `bind: address already in use` у Caddy | Порт 80/443 на ПК занят (IIS/Skype). Освободите (`Stop-Service W3SVC`) и перезапустите. |
| Работает по Wi-Fi дома, но не снаружи | Это «hairpin NAT»: проверяйте всегда из внешней сети (мобильный интернет). |

## 10. Безопасность (критично!)
Прямой хостинг открывает ваш ПК всему интернету.
- Стенд по умолчанию `DEMO_MODE=true` → **любой зашедший = ADMIN**. Перед публикацией в `.env`:
  `DEMO_MODE=false`, сильный `JWT_SECRET` (≥32 симв.), реальный пароль БД, реальные пользователи
  (см. `docs/SECURITY_ANALYSIS.md`).
- Не открывайте лишние порты; не используйте DMZ.
- Регулярно обновляйте образы (`docker compose pull`).
- Рассмотрите ограничение доступа по IP/паролю на уровне Caddy (basic auth) или переход на туннель
  с Cloudflare Access — это безопаснее прямого выставления ПК.
