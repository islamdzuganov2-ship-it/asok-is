## 1. Карточка «Качество информационной системы» (дашборд «Динамика качества»)

- В самом верху вкладки «Динамика качества» — карточка **«Качество информационной системы»**:
  интегральный показатель качества ИС с разбивкой **по кварталам** (Q1-2025…Q2-2026), в том же
  стиле, что и карточка «Качество по характеристикам во времени».
- Интегральный ряд = среднее измеримых характеристик в каждом квартале (сквозная согласованность
  с карточкой характеристик, покрыто юнит-тестом).
- Клик по точке — модалка причин изменения по кварталам.
- Реализация: `frontend/src/pages/dashboard/QualityDynamicsPage.tsx` (карточка №0),
  `frontend/src/data/mockScaleData.ts` (`DYNAMICS[*].system`). **[фронт]**

## 2. Все системы на одной диаграмме / выбор системы

- В селекторе систем добавлен пункт **«— Все системы на одной диаграмме —»**: интегральные ряды
  всех 30 ИС на одном графике (скроллируемая легенда).
- Клик по линии системы в режиме «все» — переход к детальному виду этой системы.
- Реализация: `QualityDynamicsPage.tsx` (`ALL_SYSTEMS`). **[фронт]**

## 3. Аномалии: всплывашка с причиной от МК + уведомления

- **Аномалия** = изменение к предыдущему кварталу на **≥ 12 п.п.** (`ANOMALY_THRESHOLD`),
  рост или просадка. Точки аномалий подсвечены (крупный красный маркер) на карточке качества ИС
  и графике характеристик.
- Во **всплывашке** точки аномалии — чёткое описание причины от Менеджера по качеству;
  при отсутствии — предупреждение «⚠ причина не указана».
- МК получает **уведомление в колокольчик** о необходимости заполнить причину (агрегировано
  по системам, топ-6 + сводка), ссылка ведёт на «Динамику качества»; на странице — Alert
  со счётчиком незаполненных причин.
- В модалке ввода причин аномальные кварталы помечены тегом «аномальное изменение — укажите
  причину», поле подсвечено как обязательное.
- Демо-данные: сценарные аномалии по характеристикам и целым системам
  (`CHAR_ANOMALIES`/`SYSTEM_ANOMALIES`), у половины причина предзаполнена МК
  (`DYNAMICS_REASONS`), у остальных — нет (для демонстрации уведомлений).
- Реализация: `mockScaleData.ts` (`detectAnomalies`), `QualityDynamicsPage.tsx`,
  `DynamicsModal.tsx`, `NotificationBell.tsx`, `dynamicsSlice.ts` (persist v2). **[фронт]**

## 4. Связь карточки «Заполненные профессиональные суждения» с характеристикой

- На дашборде МК карточка «Заполненные профессиональные суждения — «ИС»» имеет **чёткую связь
  с выбранной характеристикой** (той же, что в карточке «Метрики характеристики "X"»):
  заголовок дополняется тегом характеристики, список фильтруется по ней (переключатель
  «только выбранная характеристика», по умолчанию включён).
- Данные — из БД: `GET /assessments/judgments-filled` (таблица `professional_judgments`,
  связка система + период + характеристика + подхарактеристика). Для «АОКИС» засеяны суждения
  с фокусом «Функциональная пригодность». **[фронт + бэк]**
- Реализация: `FilledJudgmentsCard.tsx`, `ManagerDashboard.tsx`, сид `seed_demo.py`.

## 5. Правки мер топ-менеджером + история изменений (аудит)

- В карточке меры (дашборд топ-менеджера / реестр мер) — кнопка **«Внести правки»**
  (роли ADMIN/CTO/CEO/CIO/EXECUTIVE): название меры/риска, обоснование, ожидание от ЛПР,
  ответственный, должность, срок.
- Каждая правка пишется в **историю изменений** (кто, когда, поле, было → стало).
- История открывается **кнопкой-иконкой без текста** (иконка истории `HistoryOutlined`),
  отдельная модалка со списком правок; пустая история — «Правок ещё не было».
- Реализация: `governanceSlice.ts` (`editProposal`, `Proposal.history`, storage v2),
  `MeasureDecisionModal.tsx`. **[фронт]**

## 6. Ролевые демо-данные (моки и БД) — пересозданы

### Моки (фронт, демо-режим) **[фронт]**
- Ответственные мер — по характеристикам (кто реально чинит): разработка/эксплуатация/ИБ/
  тестирование/архитектор (`OWNER_BY_CHAR`), эскалация — по критичности ИС:
  MISSION CRITICAL → CIO, BUSINESS CRITICAL → CTO, BUSINESS OPERATIONAL → руководитель ИТ-блока.
- Меры: 47 шт (35 ожидают / 8 одобрено / 4 отклонено), срывы выполнения эскалированы с причиной,
  часть эскалаций решена топ-менеджментом (`REQUEST_MEASURES` + указание), одобренные имеют
  ссылку на СУЗ.
- Демо-ключи localStorage v2 (`asok_governance_v2`, `asok_dynamics_reasons_v2`) — старые
  нагенерированные данные удаляются при загрузке.

### БД (бэк, live-режим) **[бэк]**
- `seed_demo.py` v2: полная очистка сгенерированных оценок и пересев **детерминированными
  сценарными** данными (без random): 4 ИС × 6 кварталов × 55 метрик каталога.
  - **АБС Core** (MISSION CRITICAL) — стабильно высокое качество, инцидент P1 в Q4-2025
    («Надёжность» −30 п.п.) с восстановлением;
  - **CRM ОПК** (BUSINESS CRITICAL) — деградация 73%→51% (функциональная пригодность,
    сопровождаемость), частично незаполненные суждения → уведомления МК;
  - **HR Portal** (BUSINESS OPERATIONAL) — рост 51%→77% после программы качества,
    2 метрики «невозможно измерить» (нет базы B);
  - **АОКИС** (BUSINESS CRITICAL) — слабая «Функциональная пригодность» (связь с картой суждений §4).
- Суждения МК — ролевые тексты: факт → влияние → мера → ответственный → эскалация по критичности.
- Контур СИИ: «Скоринг-ML (СИИ)» (system_kind=AI), период Q2-2026, 16 значений
  (матрицы ошибок 1000 примеров, эталон 0.88 ± 0.10, 3 несоответствия) — таблица
  `ai_assessment_values`, не смешивается с ISO-контуром.
- Мусорные системы прошлых сессий (30 зеркал моков + QA-хлам) — мягко удалены
  (`is_deleted`, `is_active=false`); дубли по имени деактивированы.
- Демо-пользователь топ-менеджмента: `cto / Cto12345!` (роль CTO).
