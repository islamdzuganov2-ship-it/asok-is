## R3. Новая вкладка «База рисков» (`/risks`) — пополняемая, источник для LLM

> Ключевое отличие от `risk_matrices`: это **сквозной накопительный реестр знаний о рисках**,
> не привязанный к одному периоду, который LLM использует для обоснований и рекомендаций (grounding/RAG).

### R3.1 Модель данных — новая таблица `risk_base`
| Поле | Тип | Назначение |
|------|-----|-----------|
| `id` | UUID | PK |
| `code` | str (уник.) | человекочитаемый код риска (напр. R-TEST-001) |
| `title` | str | краткое название |
| `category` | str | категория (тестируемость, надёжность, безопасность, данные…) |
| `characteristic` / `subcharacteristic` | str | привязка к ISO 25010 / МК_8.1 |
| `description` | text | описание риска |
| `consequence` | text | последствие |
| `mitigation` | text | типовые меры минимизации |
| `severity` | enum (low/medium/high/critical) | критичность |
| `likelihood` | enum | вероятность |
| `triggers` | text/json | признаки/пороги срабатывания (для авто-связки с метриками) |
| `keywords` / `embedding` | json/vector | для поиска LLM (RAG) |
| `source` | enum (manual/excel/governance/llm) | происхождение записи |
| `system_id` | UUID nullable | если риск специфичен для ИС, иначе общий |
| `status` | enum (active/archived) | актуальность |
| `created_by` / `created_at` / `updated_at` | meta | аудит |

### R3.2 Функциональность вкладки
- Таблица с поиском/фильтрами (категория, характеристика, severity, статус, ИС).
- **Добавление/редактирование** записи вручную (модалка) — «которую можно дополнять».
- Импорт: из `risk_matrices` периода (кнопка «Перенести в базу») и из `.xlsx`.
- Авто-наполнение: при `Proposal.createRisk=true` (governance) создаётся черновик записи в базе.
- Версионирование/архив вместо удаления (для воспроизводимости решений LLM).

### R3.3 Использование LLM (целевое, см. [[ТЗ_LLM_MVP_v10]])
- Эндпоинт `GET /risks/search?q=...` (по keywords/embedding) → top-N релевантных рисков.
- В промпт `generate_ai_summary_task` добавляется блок «Известные риски по просевшим
  характеристикам» из базы → рекомендации **обоснованы реальной базой**, а не выдуманы.
- При формировании меры качества LLM предлагает `mitigation` из базы (reuse, без галлюцинаций).
- Post-MVP: эмбеддинги (pgvector) для семантического поиска; обратная связь — одобренные меры
  обогащают базу (`source=governance`).

### R3.4 API (черновик)
- `GET /risks` (фильтры, пагинация) · `POST /risks` · `PATCH /risks/{id}` · `POST /risks/{id}/archive`
- `GET /risks/search?q=` (для LLM) · `POST /risks/import/{period_id}` (из risk_matrices)

### R3.5 Доступ (RBAC)
- Чтение: все аутентифицированные. Создание/редактирование: `QUALITY_MANAGER`, `ADMIN`.
- Топ-менеджмент видит базу read-only как обоснование мер.

### R3.6 Навигация
- Пункт меню «База рисков» в [AppLayout.tsx](../frontend/src/components/AppLayout.tsx), роут `/risks`.

### R3.7 Acceptance R3
- Запись можно создать/отредактировать/архивировать вручную и через импорт.
- `GET /risks/search` возвращает релевантные записи; LLM-промпт включает блок известных рисков.
- Мера качества с `createRisk` порождает черновик в базе.

## Аддендум 2026-06-30 — UI/UX правки (применены в коде, проверены QA)

### A1. Паритет дашбордов «Демо ↔ LLM/реальная БД» (тепловые карты)
**Проблема:** в режиме LLM (реальная БД) дашборды строились иначе, чем в моках: имена
характеристик из Excel-импорта двоились (≈14 «характеристик» вместо 8), а аналитическая
теплокарта не получала баллы ячеек/шапки и drill-down (бэкенд их не отдавал).
**Решение:**
- Read-time нормализация имён характеристик к 8 каноническим (ISO/IEC 25010) —
  `backend/app/constants/quality_model.py::canonical_characteristic` (ё/е, регистр, префиксные
  «хвосты», синонимы: «Безопасность»→«Защищённость», «Эффективность…»→«Производительность» и др.).
- `GET /assessments/dashboard` теперь отдаёт ту же структуру, что мок `ANALYTICS_SCALE`:
  `xAxisLabels` = 8 характеристик в каноническом порядке, `characteristics` (средние по
  характеристике/подхарактеристике), `systemDetails` (баллы по каждой ячейке) → теплокарта
  `LevelHeatmap` рендерит % в ячейках/шапке и drill-down, как в демо.
- `GET /reports/executive-dashboard` нормализует характеристики к 8; в шапке — аббревиатуры
  (паритет с моком).
**Acceptance:** обе теплокарты в режиме LLM визуально идентичны демо (8 колонок, цвета по уровням,
серые ячейки «невозможно измерить», %, drill-down).

### A2. «Невозможно измерить» (нет возможности собрать данные)
**Требование:** в оценке — галочка «Невозможно измерить»; при её установке **комментарий
обязателен** (описать причину, почему данные нельзя собрать).
**Решение:**
- БД: `assessment_values.unmeasurable BOOL` (миграция `004_unmeasurable_flag`).
- API (`PUT /assessments/{id}/metrics`, `POST /assessments/{id}/values`): при `unmeasurable=true`
  комментарий обязателен (иначе 422 / запись отклоняется), `calculated_x=NULL`,
  `quality_level='Невозможно измерить'`, `val_a/val_b` очищаются.
- Полнота/`finalize`: значение «невозможно измерить» с обоснованием засчитывается как
  обработанная подхарактеристика (пара закрыта осознанно).
- Frontend (`MetricsInputPage`): колонки «Невозм. изм.» (чекбокс) + «Комментарий»
  (обязателен и подсвечивается при включённой галочке; val_a/val_b блокируются).
- Дашборды: уровень «Невозможно измерить» отображается серым (как в моках, `UNMEASURABLE`).
