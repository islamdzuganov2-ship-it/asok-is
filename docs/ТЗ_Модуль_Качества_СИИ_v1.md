# АСОК ИС — ТЗ v1: Модуль оценки качества систем искусственного интеллекта (СИИ) по ГОСТ Р 59898‑2021

> **Цель:** дополнить АСОК ИС вторым контуром оценки качества — для **систем ИИ/ML (СИИ)** —
> по национальному стандарту **ГОСТ Р 59898‑2021**, сохранив текущий контур ISO 25010 / ГОСТ Р ИСО/МЭК 25010‑2015 для классических ИС.
> Правки в код **не внесены** — документ описывает целевое состояние «к применению по команде».
> Источник модели и формул: распарсенный официальный PDF `ГОСТ Р 59898—2021` (26 с., текстовый слой).

---

## ЧАСТЬ A. Контекст и обоснование

### A1. Зачем
ISO 25010 (и его ГОСТ‑аналог) описывает качество **классического** ПО и не учитывает специфику ИИ: обучаемость, объяснимость, робастность к состязательным входам, приватность данных, контролируемость. Для банковского контура, где внедряются ML‑модели (скоринг, антифрод, OCR, чат‑боты), нужна оценка по профильному нацстандарту **ГОСТ Р 59898‑2021 «Оценка качества систем искусственного интеллекта. Общие положения»**.

### A2. Нормативная база
| Стандарт | Роль в модуле |
|---|---|
| **ГОСТ Р 59898‑2021** | Основной: модель качества СИИ, методология, критерий соответствия, метрики (разд. 8) |
| ГОСТ Р 59276‑2020 | Базовая классификация характеристик (доверие к СИИ), принципы выбора набора |
| ГОСТ Р ИСО/МЭК 25010 | Преемственность: 59898 наследует и расширяет характеристики 25010 |
| ГОСТ Р ИСО 5725‑1 | Свойства метрик: достоверность, воспроизводимость, повторяемость, показательность |

## ЧАСТЬ B. Анализ стандарта (модель качества СИИ)

### B1. Иерархия модели (4 уровня)
ГОСТ 59898 строит дерево (рис. 1) глубже, чем 25010:

```
Группа характеристик → Характеристика (по ГОСТ Р 59276) → Существенная характеристика → Субхарактеристика → Метрика(и)
```

На каждом уровне — **весовые коэффициенты** (Σ = 1) и взвешенная свёртка снизу вверх (разд. 7.2.2.4, формулы 3–8):

- субхарактеристика: `c = Σ vⱼ·mⱼ`, `Σ vⱼ = 1` (L метрик);
- характеристика: `K = Σ wᵢ·cᵢ`, `Σ wᵢ = 1` (N субхарактеристик);
- интегральный показатель качества СИИ: `Q = Σ uₖ·Kₖ`, `Σ uₖ = 1`, **Q ∈ [0,1]**, чем ближе к 1 — тем выше качество.

### B2. Полная модель качества продукта СИИ (таблица 2 стандарта)
🟦 — характеристика 25010; 🟢 — **ИИ‑специфичная субхарактеристика (нет в 25010)**.

| Группа | Характеристика (59276) | Субхарактеристики |
|---|---|---|
| Функциональность | Функциональные возможности (functionality) | функц. пригодность; функц. корректность; **согласованность (compliance)** 🟢; функц. полнота; **способность к самообучению (ability to learn)** 🟢 |
| Функциональность | Способность к взаимодействию (compatibility) | соответствие (co‑existence); функц. совместимость (interoperability); **контролируемость (controllability)** 🟢 |
| Функциональность | Уровень производительности (performance efficiency) | характер изменения во времени; характер изменения ресурсов; производительные возможности (capacity) |
| Функциональность | Мобильность (portability) | адаптируемость; простота внедрения (installability); (взаимо)заменяемость |
| Функциональность | Практичность (usability) | **понятность/объяснимость (explainability)** 🟢; изучаемость; простота использования; защищённость от ошибки пользователя; эстетика UI; доступность; **взаимодействие (collaborability)** 🟢 |
| Сопровождаемость | Сопровождаемость (maintainability) | анализируемость; изменяемость; устойчивость (stability); тестируемость; модульность; **настраиваемость (evolution)** 🟢 |
| Надёжность | Надёжность (reliability) | стабильность (maturity); отказоустойчивость; восстанавливаемость; **робастность (robustness)** 🟢 |
| Безопасность | Защищённость (security) | конфиденциальность; целостность; неотказуемость; подотчётность; подлинность; **приватность (privacy)** 🟢 |

**Итого:** 4 группы, 8 характеристик, **37 субхарактеристик**, из них **7 ИИ‑специфичных** (+ «согласованность», которой нет в 25010:2011). Набор — **представительный**: номенклатура выбирается под конкретную СИИ и фиксируется в её ТЗ (п. 7.1.4).

### B3. Критерий соответствия (разд. 7.1.3) — ключевое отличие от 25010
Качество = степень соответствия измеренного значения **базовому (baseline) / эталонной системе Sᵣ**, а не «больше — лучше». Необходимое условие соответствия:

```
m_l − ε⁻  ≤  m_f  ≤  m_l + ε⁺
```
где `m_f` — измеренное значение характеристики, `m_l` — установленное (базовое) значение, `ε⁻, ε⁺` — допустимые отклонения. Метрика **нормируется в [0,1]** (п. 7.2.2.3): чем ближе к 1 — тем ближе к базовому значению; = 1 при совпадении.

### B4. Метрики ML (разд. 8 — рекомендательные)
| Тип задачи | Метрики (формулы стандарта) |
|---|---|
| Базовые (A/B) | полнота `M1=1−A/B`; корректность `M2=1−A/B`; степень автоматизации `M4=A/B`; производительность `M6=A/T` |
| Классификация/обнаружение | accuracy `(TP+TN)/(TP+TN+FP+FN)`; precision `TP/(TP+FP)`; recall `TP/(TP+FN)`; specificity `TN/(TN+FP)`; **F‑мера** `2·Pr·Re/(Pr+Re)`; AUC‑ROC; AUC‑PRC |
| Регрессия | MSE; MAE |
| Ранжирование | NDCG (DCG/IDCG) |
| Синтез/реконструкция изображений | PSNR; SSIM |
| Время отклика | отклонение `M5` от допустимого `Tн` |

### B5. Процесс оценки (разд. 7.2, 9)
3 стадии: **подготовка** (стенд, набор данных, критерий выбросов, выбор характеристик/метрик, значимые факторы) → **тестирование и оценка** (измерение метрик, нормирование, веса, свёртка) → **анализ и интерпретация**. Дополнительно стандарт требует: паритет тестовой и эксплуатационной сред (табл. 3), выявление выбросов в данных, **экспертную группу с критерием согласованности**, требования к тестовым наборам (разд. 9).

## ЧАСТЬ D. Целевая архитектура модуля

### D1. Принцип
Не дублировать оценочный контур, а **расширить** существующие сущности минимальными миграциями. Контур СИИ отделяется от контура ИС полем `model_kind`/`system_kind`, переиспользуя периоды, значения, артефакты, экспертику, дашборды.

### D2. Модель данных (новые таблицы/поля)
```text
System
  + system_kind: ENUM('CLASSIC','AI')  -- маршрутизация на модель качества

qm_node (новая)  -- дерево модели качества (универсально для ISO25010 и 59898)
  id, model_kind ENUM('ISO25010','GOST59898'),
  level ENUM('GROUP','CHARACTERISTIC','SUBCHARACTERISTIC'),
  parent_id -> qm_node.id (nullable),
  name_ru, name_en, code, is_ai_specific BOOL, sort

metric_definition (расширение metric_catalog)
  + model_kind, qm_node_id -> qm_node (субхарактеристика),
  + metric_kind ENUM('RATIO_DIRECT','RATIO_INVERSE','ACCURACY','PRECISION',
        'RECALL','SPECIFICITY','F1','AUC_ROC','AUC_PRC','MSE','MAE',
        'NDCG','PSNR','SSIM','EXPERT_SCALE'),
  + inputs_schema JSONB   -- какие поля собирать (TP/TN/FP/FN; y/ŷ; A/B/T …)

assessment_value (расширение)
  + inputs JSONB           -- сырьё метрики (TP,FP,…; массивы для MSE/MAE)
  + baseline NUMERIC, tol_low NUMERIC, tol_high NUMERIC  -- m_l, ε⁻, ε⁺
  + normalized_x NUMERIC   -- нормировка к baseline в [0,1] (п.7.2.2.3)
  + conformant BOOL        -- попадание в допуск (п.7.1.3.3)
  + weight NUMERIC         -- вес метрики vⱼ внутри субхарактеристики

qm_weight (новая)          -- веса субхар. в хар-ке и хар-к в Q (per период/СИИ)
  period_id, qm_node_id, weight   -- Σ по уровню = 1 (валидируется)

reference_system (новая)   -- Sᵣ: эталон и baseline-значения
test_dataset (новая)       -- метаданные набора (разд.9): объём, репрезентативность,
                              критерий выбросов, паритет сред
expert_group_eval (новая)  -- групповая экспертная оценка + коэф. согласованности
```

### D3. Расчётный движок — `ai_calculation_engine.py` (новый, рядом с [calculation_engine.py](backend/app/services/calculation_engine.py))
1. `compute_metric(metric_kind, inputs)` → сырое значение (accuracy/F1/MSE/…).
2. `normalize_to_baseline(value, baseline, tol_low, tol_high)` → X∈[0,1] + `conformant` (п. 7.2.2.3, 7.1.3.3).
3. `aggregate(period)` → субхар. = Σvⱼmⱼ → хар. = Σwᵢcᵢ → **Q = ΣuₖKₖ** (формулы 3–8), с валидацией Σвесов = 1.
4. `map_to_level(Q)` — переиспользовать существующую шкалу уровней.

### D4. API — новый роутер `/api/v1/ai-assessments/*` (по образцу [assessments.py](backend/app/api/v1/endpoints/assessments.py))
- `GET  /ai-model` — дерево модели 59898 (группы→хар→субхар).
- `POST /periods` / `GET /periods` — периоды оценки СИИ (переиспользуют `AssessmentPeriod` + `model_kind`).
- `PUT  /{period}/subcharacteristics` — выбор представительного набора под СИИ.
- `PUT  /{period}/values` — ввод метрик (ML‑входы, baseline, допуски).
- `PUT  /{period}/weights` — веса уровней (валидация Σ=1).
- `POST /{period}/calculate` — нормирование + свёртка → интегральный Q.
- `POST /{period}/finalize` — завершение при полноте выбранного набора.
- `GET  /{period}/conformance-report` — отчёт соответствия критериям (по допускам).

### D5. Frontend — раздел «Оценка СИИ»
- Признак системы «СИИ» в карточке системы → открывает контур 59898 вместо ISO 25010.
- Каскад **группа → характеристика → субхарактеристика → метрика**; для ML‑метрик — ввод TP/TN/FP/FN, массивов y/ŷ (или загрузка из теста), baseline и допусков.
- Редактор весов (с проверкой Σ=1), индикатор соответствия (в допуске/вне).
- Дашборд интегрального Q + теплокарта по характеристикам (переиспользовать [LevelHeatmap.tsx](frontend/src/components/LevelHeatmap.tsx)); LLM‑интерпретация через [llm_service.py](backend/app/services/llm_service.py).
- Источник модели на фронте: новый `frontend/src/constants/aiQualityModel.ts` (по образцу [qualityModel.ts](frontend/src/constants/qualityModel.ts)).

### D6. Роли
Переиспользуются ADMIN / QUALITY_MANAGER / TEST_ANALYST. Рекомендуется ввести роль **ML/AI‑эксперт** (ввод ML‑метрик и baseline) — согласовать с [ТЗ_Ролевая_модель_v12.md](docs/ТЗ_Ролевая_модель_v12.md).

## ЧАСТЬ E. Этапы внедрения
- **E1 — MVP:** справочник модели 59898 (seed `seed_gost59898.py`), признак `system_kind=AI`, ввод субхарактеристик с метриками A/B + классификация (accuracy/precision/recall/F1), baseline+допуски, простой интегральный Q (равные веса). Переиспользует `AssessmentPeriod/Value`.

- **E3:** тестовые датасеты и выбросы, паритет сред (табл. 3), экспертная группа + согласованность, сравнение нескольких СИИ (единые шкалы, п. 7.2.2.5).

## ЧАСТЬ F. Критерии приёмки
1. Систему можно пометить как СИИ; для неё открывается контур 59898, для остальных — ISO 25010 (без смешения).
2. Каталог модели 59898 засеян: 4 группы, 8 характеристик, 37 субхарактеристик, 7 помечены ИИ‑специфичными.
3. Поддержаны ≥ метрики: A/B (прямая/обратная), accuracy, precision, recall, specificity, F1 (E1); MSE, MAE, AUC, NDCG, PSNR, SSIM (E2).
4. Для метрики хранятся baseline и допуски; рассчитываются `normalized_x` и `conformant`.
5. Веса валидируются (Σ по уровню = 1); интегральный Q ∈ [0,1] считается по формулам 3–8.
6. `finalize` доступен только при полноте выбранного представительного набора.
7. Отчёт соответствия показывает по каждой характеристике: значение, базовое, допуск, вердикт.
8. Дашборд отображает Q и теплокарту; ISO‑контур не затронут (регресс‑проверка).

## ЧАСТЬ G. Риски и открытые вопросы
- Метрики разд. 8 — **рекомендательные**: номенклатура должна быть настраиваемой под отрасль/задачу.
- Субхарактеристики «объяснимость», «самообучение», «контролируемость», «взаимодействие» не имеют жёстких формул в стандарте → **экспертная шкала** (`EXPERT_SCALE`) с обоснованием.
- baseline/эталонная система требует организационного процесса (откуда брать эталон) — согласовать с заказчиком.
- Не смешивать с контуром ISO 25010 в дашбордах и отчётах (разделять по `model_kind`).
- Согласовать роль ML‑эксперта с ролевой моделью v12.

| specificity | `TN/(TN+FP)` | SPECIFICITY | TN,FP |
| F‑мера | `2·Pr·Re/(Pr+Re)` | F1 | TP,FP,FN |
| MSE / MAE | по форм. 11/12 | MSE / MAE | массивы y, ŷ |
| AUC‑ROC / AUC‑PRC | по форм. 18/19 | AUC_ROC / AUC_PRC | кривая Se/Sp или Pr/Re |
