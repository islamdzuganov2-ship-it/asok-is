# АСОК ИС — ТЗ v12: Ролевая модель (детальная)

> Канонический источник — этот документ. Синхронизирован с кодом
> (`backend/app/api/deps.py`, `models/user.py`, фронтовые роуты/меню) и заметками Obsidian
> (`qwin/…`, `AI/АСОК_ИС…`). Канонизирует разрозненные роли в **три**.

## 1. Принципы
- **Минимум привилегий** (least privilege): каждая роль получает только необходимое.
- **Разделение обязанностей** (SoD): тот, кто фиксирует данные, не принимает решение по мере;
  тот, кто решает, не отчитывается о выполнении. Это контур управления качеством.
- **Нейминг по уровню, а не по титулу**: верхний уровень не привязан к CIO/CTO/CEO/директору.

## 2. Роли (3 канонические)

| Код роли (внутр.) | Отображаемое имя | Кто это | Уровень |
|-------------------|------------------|---------|---------|
| `ADMIN` | **Топ-менеджмент** | ЛПР верхнего уровня: видит картину качества ИТ-ландшафта, принимает решения по мерам, задаёт приоритеты и ресурсы. Нейминг намеренно без конкретной должности. Совмещает бизнес-решения и администрирование системы. | Стратегический |
| `QUALITY_MANAGER` | **Менеджер по качеству** | Формирует профессиональные суждения, ставит меры/задачи, контролирует их выполнение, ведёт базу рисков. | Тактический |
| `TEST_ANALYST` | **Аналитик** | Собирает и вводит исходные данные (val_a/val_b), грузит Excel, готовит расчёты. | Операционный |

### 2.1 Маппинг легаси-ролей → канон
В коде встречались `CTO`, `CEO`, `CIO`, `EXECUTIVE`. Все они — **верхний уровень** и
приравниваются к роли **Топ-менеджмент** (`ADMIN`-tier) по доступу. Канон — три роли выше;
легаси-коды принимаются как алиасы для обратной совместимости, новые учётки заводятся
только в трёх канонических ролях.

## 3. Матрица прав (resource × role)

Обозначения: **C** создание · **R** чтение · **U** изменение · **D**/Arch удаление/архив · «—» нет доступа.

| Ресурс / действие | Топ-менеджмент | Менеджер по качеству | Аналитик |
|-------------------|:--:|:--:|:--:|
| Управленческий дашборд | R (полный) | — | — |
| Дашборд менеджера по качеству | R | R (полный) | — |
| Аналитический дашборд | R | R | R |
| Настройки / фиче-флаги | R·U | — | — |
| Пользователи (админ.) | C·R·U·D | — | — |
| ИС (системы) | C·R·U·D | C·R | C·R |
| Метрики каталога | C·R·U | C·R | C·R |
| Оценочный период | C·R | C·R | C·R |
| Ввод val_a / val_b (ручной) | R | R·U | **C·R·U** |
| Импорт Excel | R | C | C |
| Экспорт отчётов (CSV/реестры) | R | R | R |
| Профессиональное суждение | R | **C·R** | — |
| Мера качества (постановка) | R | **C·R** | — |
| Решение по мере (одобрить/отклонить + коммент.) | **C·U** | — | — |
| Смена ответственного/срока (до решения) | **U** | — | — |
| Контроль выполнения (выполнено/не выполнено + коммент.) | R | **C·U** | — |
| База рисков | R·C·U·Arch | **C·R·U·Arch** | R |
| Реестр мер качества | R (+решение) | R | — |

> SoD-инварианты (зашиты в логику):
> - меру **создаёт** менеджер по качеству, **решение** принимает топ-менеджмент (нельзя «сам себе одобрил»);
> - **отчёт о выполнении** даёт только менеджер по качеству; топ-менеджмент видит результат, но не отмечает;
> - менять ответственного и срок до решения может только топ-менеджмент.

## 4. Контур управления качеством (governance loop)
```
Аналитик → вводит данные (val_a/val_b, Excel)
   ↓
Менеджер по качеству → проф. суждение + мера (кто ответственный, должность, срок)
   ↓
Топ-менеджмент → решение (одобрить/отклонить, может скорректировать ответственного/срок)
   ↓
Менеджер по качеству → отчёт о выполнении (выполнено/не выполнено + обязательный комментарий)
   ↓
Топ-менеджмент → видит статус-техдолга (burndown, просрочки)
```

## 5. Где это enforced
- **Backend:** `app/api/deps.py::require_role(...)` на эндпоинтах (risk_base CRUD — QM/ADMIN,
  excel-импорт — Analyst/QM/ADMIN, seed — ADMIN). `models/user.py` — коды ролей.
- **Frontend:** `App.tsx` (`RequireRole`, ролевой `DashboardRouter`), `AppLayout` (меню по роли),
  governance-логика (решение — топ-менеджмент; выполнение — QM).

### 5.1 SoD по мерам качества (уточнено и проверено QA 2026-06-30)
- **Согласование работ по мере (одобрить/отклонить) — ТОЛЬКО роль `ADMIN` (топ-менеджмент).**
  Enforced в `components/MeasureDecisionModal.tsx` (`canDecide`) и в реестре мер
  `pages/ExcelReportsPage.tsx` (колонка «Решение» скрывает кнопки для не-ADMIN).
  Прочим ролям отображается «Ожидает решения топ-менеджмента».
- **Закрытие меры / контроль выполнения (выполнено/не выполнено + обязательный комментарий) —
  ТОЛЬКО роль `QUALITY_MANAGER`.** Enforced в `MeasureDecisionModal.tsx` (`canReportExecution`).
- Инвариант сохранён: тот, кто согласовывает (ADMIN), не отчитывается о выполнении; тот, кто
  закрывает (QM), не принимает решение — «сам себе не одобрил».
- ⚠️ Меры реализованы как фронтовый governance-слой (Redux+localStorage), поэтому SoD-гейтинг —
  на UI. При выносе мер в backend (POST/PATCH governance) продублировать `require_role`.

## 6. Нейминг (правила)
- Внутренние коды НЕ меняются: `ADMIN`, `QUALITY_MANAGER`, `TEST_ANALYST` (стабильность токенов/БД).
- Человеко-читаемые подписи — централизованно (`frontend/src/constants/roles.ts`),
  чтобы не было разнобоя: **Топ-менеджмент / Менеджер по качеству / Аналитик**.
- В интерфейсе показываем подпись, а не код.

## 7. Аналитика и предложения

### Применено
1. **Канонизация в 3 роли** + маппинг легаси (CTO/CEO/CIO/EXECUTIVE → Топ-менеджмент).
2. **Централизованные подписи ролей** (единый источник, без разнобоя в строках).
3. **SoD усилён**: контроль выполнения — только менеджер по качеству; решение — только топ-менеджмент.
4. **Нейминг верхнего уровня** без привязки к должности — «Топ-менеджмент».

### Предложения на будущее (по команде)
- **Разделить `ADMIN`** на «Топ-менеджмент» (бизнес-решения) и «Системный администратор»
  (пользователи, фиче-флаги, сидинг) — чистый SoD, если появится выделенный админ.
- **Роль «Ответственный исполнитель»** (read-меры + отметка о факте по своим задачам), если
  исполнение делегируется вне менеджера по качеству.
- **Журнал ИБ-событий по ролям** (см. backlog S10) — кто и какое действие совершил.
- LDAP/SSO-маппинг групп каталога на эти три роли.
