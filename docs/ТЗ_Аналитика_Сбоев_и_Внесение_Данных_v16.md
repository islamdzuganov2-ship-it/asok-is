### T-36. Обязательные поля карточки сбоя + связь с мерой (модель, миграция) **[бэк]**
Добавить в `TechIncident` и схемы поля (все — обязательные при ручном вводе, кроме `linked_measure_id`):
- `root_cause` — **корневая причина** (уже есть → сделать обязательным на вводе).
- `admission_cause` (Text) — **причина допущения** («почему сбой стал возможен / что не сработало в контроле»).
- `responsible_unit` (String 255) — **виновный: направление производства**, допустившее возникновение.
- `preventive_measures` (Text) — **меры по неповторению**.
- `linked_measure_id` (UUID, nullable) — **связь с мерой по улучшению качества** (governance-предложение/`Proposal`); опциональна (мера может появиться позже).
- Миграция БД (Alembic, номер по порядку). Валидация обязательности — в `service.create/update` (доменные `ValidationError`), обязательность зависит от `source`: для `manual` — все поля обязательны; для `import/itsm` — мягкая (лог недостающих в `description`), т.к. внешние источники неполны.
- **Приёмка:** POST без `admission_cause/responsible_unit/preventive_measures` для `source=manual` → 422/400 с понятным сообщением; существующие записи не ломаются (nullable + бэкофилл пустыми).

### T-42. Связка «сбой ↔ мера по улучшению качества» — масштабируемый UX **[фронт]**
Требование: не перегружать при 100+ систем и 1000+ сбоев; интуитивно.

Принятое решение (двухуровневый умный подбор вместо «плоского списка на 1000 мер»):
1. В карточке сбоя поле «Связанная мера по улучшению качества» — `Select` **со скоупингом по умолчанию**: показываются меры (`Proposal`) только той же **системы** и той **характеристики**, что соответствует первопричине сбоя (использовать существующий маппинг `CATEGORY_TO_CHARACTERISTIC`). Это сужает 1000 → обычно < 20.
2. Поиск по названию/риску снимает скоуп (`showSearch`, «искать среди всех мер системы / всех систем»). Пункт показывает: риск-заголовок · характеристика · статус.
3. Обратная связка (масштабируемая): мера в «Реестре мер» показывает **все сбои** своей (система + первопричина→характеристика) — агрегирование по корневой причине, чтобы одна мера закрывала множество сбоев без ручной привязки каждого.
4. Опция «Меры ещё нет → создать меру по этому сбою» — сид governance-`Proposal` из фактуры сбоя (система, характеристика, первопричина, корневая причина). (Фаза 2.)

- **Приёмка:** из карточки сбоя можно за 1–2 клика привязать релевантную меру без прокрутки тысяч; поиск находит любую; в реестре мер видно связанные сбои.

### T-43. Загрузка Excel/CSV техсбоев (нестандартизированные поля) + маппинг + подсказка **[бэк+фронт]**
- Загрузка `.xlsx/.csv` со сбоями из внешнего ITSM/ЕХД/DWH, поля которого **не совпадают** с нашими.

- **Подсказка по формату** — Приложение F.2 (обязательные/опциональные поля, синонимы заголовков, форматы дат, единицы MTTR).
- **Приёмка:** можно загрузить файл с произвольными заголовками, сопоставить колонки, увидеть предпросмотр и результат импорта; ошибочные строки не роняют весь импорт.

## Блок D. Обязательность для неизмеримых метрик

### T-55. Неизмеримая метрика обязана иметь причину + проф. суждение + меры **[бэк+фронт]**
- Правило: метрика со статусом «Невозможно измерить» (`score < 0`) **обязана** иметь: (1) причину невозможности измерения, (2) профессиональное суждение, (3) хотя бы одну меру.
- Backend: при финализации периода — валидация, что для каждой `unmeasurable`-метрики заполнены причина+суждение и существует связанная мера; иначе `ValidationError` со списком «чего не хватает».
- Frontend (`MetricsInputPage`/`ExpertReviewPage`): визуальное принуждение — неизмеримую метрику нельзя провести дальше без причины/суждения/меры; подсветка незаполненного.
- **Приёмка:** попытка финализировать период с неизмеримой метрикой без причины/суждения/меры блокируется с понятным перечнем.

## Блок E. Корректность карточек тепловой карты характеристик

### T-56. Карточки строго по (система, характеристика) — без переиспользования мер/резюме **[фронт]**
Проблема (заказчик): при клике по характеристике в карточке показывается **системное** резюме вида «Интегральная оценка качества — 31%… Наиболее просевшая характеристика — Функциональная пригодность (17%)…» — оно одинаково для разных характеристик и не относится к выбранной.

Причина в коде: `aiSummary` формируется на уровне **системы** (`mockScaleData.ts:190`, `ExecutiveDashboard.tsx:85`) — «интегральная оценка + самая слабая характеристика». Если такой текст попадает в карточку **конкретной характеристики** — это неверно.

Задача:
- Пройти по всем местам, где карточка/модалка **характеристики** (клик по ячейке/сектору теплокарты: `DashboardPage.charDetail`, `ManagerDashboard`, `ExecutiveDashboard`) — убедиться, что показываются **резюме, меры и подхарактеристики именно этой (система × характеристика)**, а не системный `aiSummary`/чужие меры.
- Резюме характеристики генерировать из её собственной фактуры: балл характеристики + её просевшие подхарактеристики + рекомендация по **этой** характеристике (`REC[char]`), с именем системы. Никакого «наиболее просевшая характеристика X» внутри карточки другой характеристики.
- Меры в карточке характеристики — только те, у кого `norm(characteristic)` совпадает (уже так в `ManagerDashboard`, но проверить `charMeasures`/`MeasureDevelopmentPanel` на утечки между характеристиками и системами).
- **Приёмка:** для одной ИС карточки разных характеристик содержат **разные**, релевантные тексты/меры; ни в одной карточке характеристики нет системного «Интегральная оценка…»; меры одной характеристики не появляются в другой.

## Приложение F. Форматы файлов и подсказки для загрузки

### F.1. Загрузка оценок (вкладка «Загрузка оценок», T-50)
**Типы файлов:** `.xlsx` (Excel, первый лист) или `.csv` (UTF-8, разделитель `;`). Первая строка — заголовки.

**Обязательные колонки** (заголовки — точно или из синонимов):
| Наша колонка | Синонимы заголовка | Данные | Формат |
|---|---|---|---|
| `system` | Система, ИС, Информационная система, System | Название ИС | текст, как в реестре систем |
| `period` | Период, Квартал, Period | Период оценки | `Q1-2026` / `2026-Q1` / `01.2026` |
| `characteristic` | Характеристика, Characteristic | Характеристика ISO 25010 | из 8 названий модели |
| `metric` | Метрика, Подхарактеристика, Metric | Подхарактеристика/метрика | из модели качества |
| `value_a` | A, Числитель, Факт | Числитель формулы | число |
| `value_b` | B, Знаменатель, База | Знаменатель формулы | число > 0 (0/пусто → «невозможно измерить») |

**Опциональные:** `comment` (Комментарий/Обоснование), `expert_comment` (Проф. суждение), `formula` (DIRECT/INVERSE — иначе берётся из модели).

**Правила:** значение метрики считается по формуле модели (DIRECT `A/B`, INVERSE `1−A/B`); пустой/нулевой `B` → «невозможно измерить» (требует причину+суждение+меру, см. T-55). Строки с неизвестной системой/характеристикой/метрикой — отклоняются с указанием причины, остальные принимаются. Шаблон скачивается кнопкой «Скачать шаблон» (заголовки + 1 пример).

### F.2. Загрузка техсбоев (вкладка «Загрузка ТС» / дашборд сбоев, T-43/T-51)
**Типы файлов:** `.xlsx` / `.csv` (UTF-8, `;`). Первая строка — заголовки. Поля из ITSM/ЕХД/DWH **не обязаны** совпадать с нашими — при загрузке открывается **мастер сопоставления колонок** (автоугадывание по синонимам ниже, ручная корректировка).

**Обязательные:**
| Наша колонка | Синонимы | Формат |
|---|---|---|
| `system_name` | Система, ИС, CI, Сервис, Affected CI, System | текст |
| `title` | Краткое описание, Тема, Summary, Subject | текст |
| `occurred_at` | Дата возникновения, Открыт, Opened, Created, Start | `ДД.ММ.ГГГГ [ЧЧ:ММ]` или ISO `YYYY-MM-DD` |
| `category` | Первопричина, Категория, Root cause, Cause, Тип | текст → маппинг на RELEASE/INFRASTRUCTURE/PERFORMANCE/NETWORK/POWER/OTHER |

**Опциональные:**
| Наша колонка | Синонимы | Формат |
|---|---|---|
| `severity` | Критичность, Priority, Severity, Приоритет | critical/high/medium/low (или P1..P4 → маппинг) |
| `resolved_at` | Дата восстановления, Закрыт, Resolved, Closed, End | дата/время (пусто → сбой открыт) |
| `root_cause` | Корневая причина, RCA, Root cause description | текст |
| `admission_cause` | Причина допущения | текст |
| `responsible_unit` | Виновное направление, Подразделение, Assignment group | текст |
| `preventive_measures` | Меры по неповторению, Corrective actions | текст |
| `release_ref` | Релиз, Версия, Change, RFC | текст |

**Правила распознавания:**
- **Даты:** принимаются `ДД.ММ.ГГГГ`, `ДД.ММ.ГГГГ ЧЧ:ММ`, ISO-8601. Некорректная дата возникновения → строка отклоняется.
- **MTTR** не загружается напрямую — вычисляется как `resolved_at − occurred_at` (часы). Если в файле есть готовый MTTR — игнорируется (единая методика расчёта).
- **Категория/первопричина:** сопоставляется по синонимам; нераспознанное значение → `OTHER` + исходный текст в `category_custom`.
- **Приоритет:** `P1/P2/P3/P4` → `critical/high/medium/low`.
- **Дедупликация:** по (`system_name` + `title` + `occurred_at`) — повтор помечается и не создаёт дубль.
- Для `source=import` обязательные поле-правила T-36 **мягкие** (недостающее — в отчёт, не блокирует), т.к. внешние данные неполны; заполнение обязательных полей — при дальнейшей ручной доработке карточки.

**Информационная подсказка в UI:** при открытии загрузки показывать свёрнутую памятку с этой таблицей, примером заголовков и кнопкой «Скачать шаблон ТС».
